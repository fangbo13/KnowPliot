## 二、灾难级问题展示

### 2.1 CRIT-001：SSE fetch 无法取消 — 3000并发下服务器雪崩

![流式输出活动状态](../../screenshots/v3.4_stream_active.png)

**现象**：
- 用户在流式输出期间切换 Session，旧的 SSE POST 连接继续活跃
- Chrome DevTools Network tab 显示旧连接未被取消

**根因**：
- `fetch()` 调用（[chatStore.ts:233](../../frontend/src/store/chatStore.ts#L233)）缺少 `AbortController.signal` 参数
- 一旦请求发起，客户端无法终止 → 服务器端的 Django worker 线程被阻塞

**3000 并发雪崩推演**：

```
用户行为：平均每分钟切换 2-3 次 Session
每次切换 → 旧流式请求继续运行 5-15 秒
3000 用户 × 3次切换/分 × 10秒平均流式时长 = 90,000 个僵尸 SSE 连接

Django 默认配置：8 worker threads (runserver) 或 ~100 gunicorn workers
90,000 僵尸连接 vs 100 workers → 新合法请求被拒绝 → 雪崩

更糟的是：Celery worker 也会被拖慢（Redis 任务堆积）
RAG pipeline 每次调用 ~3-5秒 → 3000 × 每分钟1次 = 每分钟3000个任务
Celery -c 4 只能处理 4个并发任务 → 任务堆积 → 用户等待更长 → 更多切换 → 更多僵尸连接
→ **恶性循环 → 系统崩溃**
```

### 2.2 CRIT-002：Session 切换竞态 — 流式数据污染新 Session

![单Session流式状态](../../screenshots/v3.4_single_session_stream.png)

**现象**：
- `setActiveSession(id)` 清空 `messages` 但不重置 `isStreaming`
- 旧流的 `finishStreamingMessage()` 使用旧 session ID，但内容追加到当前 session 的 messages 数组

**数据污染推演**：

```
步骤1: Session A 发送消息 → isStreaming=true → 流式开始
步骤2: 用户切换到 Session B → setActiveSession(B) → messages=[] → isStreaming仍=true
步骤3: 旧流 reader 继续执行 → 收到 done 事件
步骤4: finishStreamingMessage(msgId, A.id) → 将 A 的助手内容追加到 state.messages
步骤5: state.messages 现已清空并指向 Session B → A的内容出现在 B的显示中
→ **数据错乱：用户看到不属于当前 Session 的内容**

更严重的是：isStreaming=true → 用户无法在 Session B 中发送新消息
→ **状态死锁：10-15秒内用户完全无法操作**
```

## 三、高严重问题展示

### 3.1 HIGH-004：Sessions API 无分页 — 9993字节单次全量返回

![Sessions API载荷](../../screenshots/v3.4_sessions_payload.png)

**实测数据**：
- 42 个 Session 返回在单次 9993-byte 响应中
- `pagination_class = None`
- `isArray=true, hasPagination=false`

**3000 并发推演**：
```
每个用户 50 个 Session × ~24KB 响应体
3000 用户 × 24KB = 72MB/次刷新 × 每分钟2-3次 = 216MB/分钟传输

数据库查询：ChatSession.objects.filter(user=request.user).order_by('-updated_at')
3000 个并行 SELECT → PostgreSQL 连接池耗尽

N+1 查询（MessageSerializer.citations SerializerMethodField）：
100条消息 × 1个额外查询/条 = 100+ 查询/请求
3000 × 100 = 300,000 个额外查询 → **数据库崩溃**
```

### 3.2 HIGH-003：无消息虚拟化 — 50轮对话前端崩溃

![4000字符边界输入](../../screenshots/v3.4_4000char_input.png)

**推演**：
```
50 轮对话 = 100 条消息
每条 MessageBubble ≈ 20-50 DOM 节点（Card + ReactMarkdown + citations）
100 × 40 = 4000 DOM 节点
每个 ReactMarkdown 完整解析 ≈ 2-5ms
100 × 5ms = 500ms 纯渲染时间（不含 React reconciliation）

页面加载时（loadMessages API）：
500KB JSON payload → 解析时间 > 100ms → 4000 DOM 节点创建 > 1s
→ **页面卡死 1-3 秒**

如果用户继续对话到 100 轮：
200 条消息 = 8000 DOM 节点 → React reconciliation > 5s
→ **浏览器标签页崩溃（内存溢出）**
```

### 3.3 HIGH-002：删除流式 Session — 后台流继续运行

![Modal覆盖侧边栏](../../screenshots/v3.4_modal_blocking_sidebar.png)

**现象**：
- `handleDeleteSession` 调用 `resetSession()` → `isStreaming: false`
- 但旧流 reader 的 `reader.read()` 循环仍在 async 上下文中执行
- 删除后服务器已无该 Session → 旧流可能收到 404 → catch 块再次 set `isStreaming: false`
- 双重 `isStreaming: false` 设置可能导致 React 状态更新冲突

## 四、架构缺失与风险说明

### 4.1 SSE 流式生命周期管理 — 完全缺失

当前 SSE 流式输出的生命周期管理完全依赖 `sendMessage` 方法内的 inline 代码，没有：

| 缺失机制 | 当前状态 | 3000并发风险 |
|----------|----------|-------------|
| AbortController / AbortSignal | ❌ 无 | 服务器线程泄漏 → 雪崩 |
| Session ID 验证（finishStreamingMessage） | ❌ 无 | 数据污染 → 用户看到错误内容 |
| isStreaming 状态守卫（Session切换） | ⚠️ 不完整 | 状态死锁 → 10-15秒无法操作 |
| 流式状态机（idle→connecting→streaming→completing） | ❌ 无 | 逻辑分散 → 难以维护 |
| 发送锁定（防双击） | ❌ 无 | 重复请求 → 资源浪费 |

### 4.2 可扩展性架构 — 3000并发无法支撑

| 缺失机制 | 当前状态 | 3000并发风险 |
|----------|----------|-------------|
| Sessions API 分页 | ❌ 无 (pagination_class=None) | 72MB/分钟传输 + DB全量查询 |
| Messages API 分页 | ❌ 无 | 500KB+单次响应 + N+1查询 |
| 前端消息虚拟化 | ❌ 无 | 4000+ DOM节点 → 前端崩溃 |
| SSE token批量渲染 | ❌ 无 | FPS<20 → 用户误判卡死 |
| SSE 端点速率限制 | ❌ 无 (@api_view不继承) | API成本失控 → DashScope无上限 |
| 会话记忆管理 | ⚠️ 后端8轮/前端全量 | 用户期望与AI上下文断裂 |
| 跨标签状态同步 | ❌ 无 | 多标签操作 → 状态不一致 |

### 4.3 会话记忆体系落差

```
当前状态：
┌───────────────────────────────────────────┐
│  前端：显示全部历史（50+轮对话完整可见）       │
│  后端：只取最近8轮/16条消息做RAG context     │
│  用户期望：AI应记得所有对话内容               │
│  AI实际：只能参考最近8轮 → 早期内容"遗忘"     │
│  → 用户困惑："我明明说了，AI怎么不记得？"     │
└───────────────────────────────────────────┘

目标架构（三层会话记忆）：
┌───────────────────────────────────────────┐
│  短期记忆：最近8轮 → 直接传入LLM context     │
│  中期摘要：AI自动总结早期要点 → 1-2轮压缩     │
│  长期索引：关键词提取 → pgvector语义检索       │
│  → AI能通过摘要引用早期关键信息               │
│  → 前端显示"AI可参考最近8轮+摘要"提示         │
└───────────────────────────────────────────┘
```

## 七、结论与建议

### 7.1 立即行动（P0）

**系统当前不可支撑 3000 人并发**。两个灾难级 Bug（CRIT-001/002）会导致：
- 服务器线程耗尽 → 雪崩
- 流式数据污染 → 数据错乱
- 状态死锁 → 用户完全无法操作

**建议**：在 1-3 天内完成 P0 修复（AbortController + Session切换竞态 + 发送防抖），然后重新进行并发测试。

### 7.2 系统性改造（P1）

6 个 HIGH 问题涉及 API 分页、前端虚拟化、token 渲染、滑动窗口透明化。每个问题都直接影响 3000 并发下的系统稳定性。

**建议**：P0 修复后，在 1-2 周内完成 P1 修复，然后进行 3000 并发模拟测试（使用 k6 或 Locust）。

### 7.3 长期演进（P3）

三层会话记忆架构、跨标签同步、流式状态机是系统长期竞争力的基础。当前缺失不会导致崩溃，但会影响核心业务价值（AI对话质量）。

**建议**：在系统稳定运行后（P0+P1 修复验证），规划 P3 架构演进。
