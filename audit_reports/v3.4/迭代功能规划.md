## P0-CRIT：流式生命周期修复（1-3 天）

> **目标**：解决 CRIT-001 和 CRIT-002 — SSE fetch 无法取消 + Session 切换竞态
> **影响**：3000 并发下会导致雪崩（服务器线程耗尽）和数据错乱（旧流污染新 Session）

### P0-CRIT-1：为 SSE fetch 添加 AbortController

**涉及文件**：`frontend/src/store/chatStore.ts`

**具体修改**：
1. 在 `ChatState` 接口新增 `currentAbortController: AbortController | null`
2. `sendMessage` 方法中，在 `fetch()` 调用前创建 `AbortController`：
   ```typescript
   const abortController = new AbortController();
   set({ currentAbortController: abortController });
   const response = await fetch(`/api/v1/chat/sessions/${sessionId}/send/`, {
     method: 'POST',
     headers: { 'Content-Type': 'application/json', Authorization: `Bearer ${token}` },
     body: JSON.stringify({ content }),
     signal: abortController.signal,  // ← 新增
   });
   ```
3. 在 `finishStreamingMessage` 中清理：`set({ currentAbortController: null })`
4. 在 catch 块中清理：`set({ currentAbortController: null })`

**验证**：
- 发送消息 → 流式期间切换 Session → Network tab 中旧 SSE 连接应显示为 cancelled
- 无 console 错误（fetch abort 不应抛出 Error 到 catch 块，只需处理 AbortError）

### P0-CRIT-2：Session 切换时中止流式并重置状态

**涉及文件**：`frontend/src/store/chatStore.ts`

**具体修改**：
1. `setActiveSession` 方法改为：
   ```typescript
   setActiveSession: (id) => {
     const { currentAbortController } = get();
     if (currentAbortController) {
       currentAbortController.abort();  // 中止旧流
     }
     set({
       activeSessionId: id,
       messages: [],
       sendError: null,
       isStreaming: false,              // ← 新增：重置流式状态
       streamContent: '',               // ← 新增：清除残留内容
       citations: [],                   // ← 新增：清除残留引用
       thinkingPhase: 'connecting',     // ← 新增：重置思考阶段
       connectionStatus: 'idle',        // ← 新增：重置连接状态
       currentAbortController: null,    // ← 新增：清除控制器
     });
   },
   ```
2. `resetSession` 方法同样添加 abort + 清理逻辑
3. `finishStreamingMessage` 验证 session ID：
   ```typescript
   finishStreamingMessage: (messageId, sessionId) => {
     const { activeSessionId } = get();
     // 如果当前活跃 Session 与流式完成的 Session 不匹配，丢弃数据
     if (activeSessionId && activeSessionId !== sessionId) {
       set({ isStreaming: false, streamContent: '', citations: [], currentAbortController: null });
       return;  // 丢弃旧流的残留数据
     }
     // ...原有逻辑...
   }
   ```
4. 所有 `clearAllTimers()` 调用点增加：`set({ currentAbortController: null })`

**验证**：
- 发送消息 → 流式期间切换到另一个 Session → 等待 3 秒 → 新 Session 不应出现旧流内容
- 切换后应能在新 Session 中立即发送消息（isStreaming 已重置为 false）

## P0-HIGH：发送防抖锁定（1 天）

> **目标**：解决 HIGH-001 — 发送按钮快速双击绕过 isStreaming 守卫

### P0-HIGH-1：添加 useRef 发送锁

**涉及文件**：`frontend/src/pages/ChatPage.tsx`

**具体修改**：
1. 在 ChatPageContainer 中新增 `sendLockRef`：
   ```typescript
   const sendLockRef = useRef(false);
   ```
2. 修改 `handleSend`：
   ```typescript
   const handleSend = () => {
     if (!inputValue.trim() || isStreaming || sendLockRef.current) return;
     sendLockRef.current = true;
     sendMessage(inputValue.trim()).finally(() => {
       sendLockRef.current = false;
     });
     setInputValue('');
     inputRef.current?.focus();
   };
   ```
3. 确保 `sendMessage` 的 `isStreaming` 守卫 + `sendLockRef` 双重保护

**验证**：
- 快速双击发送按钮 → Network tab 中只有 1 个 POST /send/ 请求
- 双击后 UI 只显示 1 条用户消息

> **目标**：解决 HIGH-003/004/005/006 — 无分页、无虚拟化、token 渲染、滑动窗口不一致

**涉及文件**：`backend/apps/chat/views.py`、`frontend/src/api/chat.ts`、`frontend/src/layout/AppLayout.tsx`

### P1-2：Messages API 分页 + 前端虚拟化

**涉及文件**：`backend/apps/chat/views.py`、`frontend/src/store/chatStore.ts`、`frontend/src/pages/ChatPage.tsx`

### P1-3：SSE token 批量渲染优化

**涉及文件**：`frontend/src/store/chatStore.ts`

**具体修改**：
1. 在 sendMessage 中添加 token 缓冲区：
   ```typescript
   let tokenBuffer = '';
   let renderScheduled = false;
   
   // 在 token event 处理中：
   tokenBuffer += data.token || '';
   if (!renderScheduled) {
     renderScheduled = true;
     requestAnimationFrame(() => {
       get().updateStreamContent(tokenBuffer);
       renderScheduled = false;
     });
   }
   ```
2. 或使用 `setTimeout(50ms)` 代替 `requestAnimationFrame`（更简单的实现）

**验证**：发送长回复问题 → Chrome Performance tab → FPS > 30 → React commit 数减少 80%

## P3：架构演进（1-3 月）

### P3-1：三层会话记忆架构

> 解决 HIGH-006 — 后端滑动窗口 vs 前端全量加载不一致

**架构设计**：
1. **短期记忆**（当前）：最近 8 轮完整对话历史（直接传入 LLM context）
2. **中期摘要**：AI 自动总结早期对话要点，压缩为 1-2 轮摘要内容（通过 Celery 异步任务）
3. **长期索引**：关键词/实体/意图提取 → 存入 pgvector 索引 → 语义检索补充上下文

**涉及文件**：新建 `backend/apps/chat/summarizer.py`、修改 `views.py` 的 history 构建、前端添加"上下文窗口"指示器

**验证**：20 轮对话 → 问 AI 早期内容 → AI 能通过摘要引用关键信息 → 用户体验提升

### P3-2：跨标签状态同步

**涉及文件**：`frontend/src/store/chatStore.ts`、新建 `frontend/src/hooks/useBroadcastChannel.ts`

**架构设计**：
1. 使用 `BroadcastChannel API` 在标签间同步关键操作（Session 创建/删除/重命名）
2. 标签 A 创建 Session → broadcast → 标签 B 的 `loadSessions()` 自动刷新
3. 对 chat 消息不做实时同步（每个标签独立管理自己的对话流）

**验证**：标签 A 创建新 Session → 标签 B 侧边栏自动更新 → 删除同理

### P3-3：流式生命周期状态机

> 将 sendMessage 内的流式逻辑从 inline 代码重构为独立的状态机

**架构设计**：
```typescript
class StreamLifecycleManager {
  state: 'idle' | 'connecting' | 'streaming' | 'completing' | 'error';
  abortController: AbortController | null;
  
  start(sessionId, content): Promise<void>;
  abort(): void;         // 清理所有状态
  complete(msgId): void; // 完成流式
  onError(err): void;    // 错误处理
}
```

**验证**：所有流式操作通过 StreamLifecycleManager → 无竞态 → 状态转换有完整日志

### P3-4：JWT Refresh Token Rotation

**涉及文件**：`backend/apps/auth/`、`frontend/src/auth/AuthProvider.tsx`、`frontend/src/api/client.ts`

**具体修改**：
1. 后端：添加 refresh token endpoint + rotation 机制
2. 前端：Axios interceptor 在 access token 过期前自动 refresh
3. SSE fetch 中的 token 也需要 refresh 机制

**验证**：access token 5 分钟过期 → 自动 refresh → SSE 流不中断
