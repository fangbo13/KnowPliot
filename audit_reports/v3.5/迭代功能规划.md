## P0 — 回归失败 + 应用崩溃级修复（必须在 V3.6 首优先级完成）

### P0-1: HistoryPage 日期分组统一（v3.5-HIGH-001）

| 字段 | 内容 |
|------|------|
| **问题** | HistoryPage.tsx:36-53 有独立的 `getDateGroup()` 未导入 `dateGroup.ts` |
| **修复方案** | 1. 将 HistoryPage 的 `getDateGroup()` 替换为导入 `dateGroup.ts` 的 `getDateGroupKey()` + `getGroupLabel()`；2. 替换 `GROUP_ORDER` 为 `computeGroupOrder()`；3. 修复硬编码 `'昨天'` 为 i18n 调用 |
| **预估工时** | 2h |
| **关键文件** | `frontend/src/pages/HistoryPage.tsx`, `frontend/src/utils/dateGroup.ts` |

### P0-2: loadSessions() 冗余调用消除（v3.5-HIGH-002）

| 字段 | 内容 |
|------|------|
| **问题** | `finishStreamingMessage` 每条消息后调用 `loadSessions()` |
| **修复方案** | 从 `finishStreamingMessage` 移除 `loadSessions()` 调用；仅在真正需要刷新的场景调用（创建新 Session、删除 Session、重命名 Session） |
| **预估工时** | 1h |
| **关键文件** | `frontend/src/store/chatStore.ts:552` |

## P1 — UI 细节 + 性能退化修复

### P1-1: allMessages 裁剪上限（v3.5-MED-001）

| 字段 | 内容 |
|------|------|
| **问题** | `allMessages` 数组无硬性上限，长对话内存线性增长 |
| **修复方案** | 设置 `MAX_ALL_MESSAGES = 100`（50 轮上限），`finishStreamingMessage` 和 `addMessage` 中当 `allMessages.length > MAX_ALL_MESSAGES` 时裁剪最旧消息 |
| **预估工时** | 2h |
| **关键文件** | `frontend/src/store/chatStore.ts` |

### P1-2: sendError 路径统一（v3.5-MED-002）

| 字段 | 内容 |
|------|------|
| **问题** | 4 种不同的 sendError 设置模式，session 创建失败用原始字符串而非 error key |
| **修复方案** | 统一所有 sendError 路径使用 error key（`error_auth`, `error_server`, `error_network`, `error_session`），UI 层统一处理 |
| **预估工时** | 2h |
| **关键文件** | `frontend/src/store/chatStore.ts:300, 308, 403-406, 450-453` |

### P1-3: computeRounds 增量优化（v3.5-MED-003）

| 字段 | 内容 |
|------|------|
| **问题** | 每条消息完成后从头遍历全部消息计算 rounds |
| **修复方案** | 缓存 rounds 计算结果，仅在 messages 数组变化时重算；或增量追加新轮次而非全量重算 |
| **预估工时** | 3h |
| **关键文件** | `frontend/src/store/chatStore.ts:124-145, 538` |

### P1-4: error_generic 错误 i18n（v3.5-MED-004）

| 字段 | 内容 |
|------|------|
| **问题** | 快速发送触发 `error_generic` 原始错误显示 |
| **修复方案** | 1. 添加 i18n key `error_generic` 的中文/英文翻译；2. UI 层对 error key 做统一友好提示 |
| **预估工时** | 1h |
| **关键文件** | `frontend/src/store/chatStore.ts`, `frontend/src/i18n/` |
