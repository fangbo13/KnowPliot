## 1. 角色定义与边界

### 1.1 三级角色体系

| 角色 | 域 | 核心职责 | 绝对禁止 |
|------|-----|---------|---------|
| **HR（知识库管理者）** | 内容域 | 管理文档分类、上传/编辑/删除/重新索引文档、管理答案模板、查看内容反馈分析、创建/编辑入职流程模板 | ❌ 不能管理用户账号；❌ 不能分配角色；❌ 不能修改系统配置；❌ 不能查看系统审计日志（仅内容域日志）；❌ 不能停用账号 |
| **Admin（系统管理者）** | 系统域 | 管理用户账号（创建/编辑/停用）、分配/撤销角色、管理 ServiceLine/OfficeLocation 配置、查看全部审计日志、系统健康仪表盘、管理入职流程批量分配、必要时介入内容操作 | ⚠️ 无域限制 — Admin 是 HR 的超集，可执行所有 HR 操作，但应委托日常内容任务给 HR |
| **Employee（标准员工）** | 基础域 | 使用 AI 聊天、查看个人入职进度、提交反馈、编辑个人偏好 | ❌ 不能进入知识库管理界面；❌ 不能查看他人入职进度；❌ 不能管理任何内容或系统资源 |

### 1.2 角色解析逻辑

```
if user.has_role('admin'):
    → 全量访问（内容轨 + 系统轨）
elif user.has_role('hr'):
    → 仅内容轨访问
else:
    → Employee 标准访问（聊天 + 个人偏好）
```

**超集关系**：Admin 角色 **隐含** HR 权限。但每次操作记录所用角色（`role_used` 字段），便于审计溯源。

> [来源: V3.8/综合审计报告_V3.8.md §权限体系] — 当前仅有 `is_hr_admin` 布尔字段，无角色模型

## 7. 前端路由守卫设计

### 7.1 RoleGuard 组件接口

```typescript
// frontend/src/components/RoleGuard.tsx
interface RoleGuardProps {
  requiredRole: 'hr' | 'admin';
  requiredPermission?: string;  // 可选，细粒度权限
  children: ReactNode;
  fallback?: ReactNode;         // 无权限时展示
}

// 使用示例
<Route path="admin/knowledge" element={
  <RoleGuard requiredRole="hr">
    <KnowledgeBasePage />
  </RoleGuard>
} />
<Route path="admin/system" element={
  <RoleGuard requiredRole="admin">
    <SystemAdminPage />
  </RoleGuard>
} />
```

### 7.2 AuthProvider 变更

```typescript
// 现有（V3.8）
interface User {
  is_hr_admin: boolean;  // ❌ 单一布尔，无法区分 HR/Admin
}

// V4.0 目标
interface User {
  roles: string[];        // ✅ ['hr'] 或 ['admin'] 或 ['hr', 'admin'] 或 []
  permissions: string[];  // ✅ ['document.create', 'category.read', ...]
}
```

> [来源: V3.8/开发变更日志汇总_V3.8.md §前端变更] — AuthProvider.tsx 用 is_hr_admin 布尔
