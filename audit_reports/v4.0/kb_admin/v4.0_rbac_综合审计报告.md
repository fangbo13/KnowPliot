## 1. 当前权限体系脆弱性分析

### 1.1 脆弱性评级：🔴 高危（2/10）

> [来源: V3.8/Bug清单汇总_V3.8.md §安全漏洞评分] — V3.8 已识别多项安全漏洞，本报告在此基础上进行深度脆弱性分析

| 维度 | 评分 | 依据 |
|------|------|------|
| **粒度** | 0/10 | 单一布尔 `is_hr_admin`，无资源-操作级权限控制。`IsHROrAdmin` 仅检查 `is_hr_admin OR is_superuser` [来源: V3.8/开发变更日志汇总_V3.8.md §permissions.py] |
| **可审计性** | 2/10 | 仅记录 11 种内容操作，无角色分配/用户管理/配置变更日志。AuditLog.ACTION_CHOICES 缺 `role_assign`, `user_create`, `config_change` 等 [来源: V3.8/项目功能模块汇总_V3.8.md §审计模块] |
| **可管理性** | 1/10 | 无角色分配 API，`is_hr_admin` 只能通过 Django Admin 或直连数据库设置。无审计追踪 |
| **信息泄露** | 3/10 | `is_hr_admin` 在 `UserSerializer` 公开暴露 [来源: V3.8/综合审计报告_V3.8.md §信息泄露]，任何登录用户可看管理员名单 |
| **防御深度** | 2/10 | `CategoryListView` 仅用 `IsAuthenticated` [来源: V3.8/Bug清单汇总_V3.8.md §权限漏洞]，前端无角色路由守卫 |

### 1.2 关键脆弱性链

```
is_hr_admin 布尔 → 无粒度控制 → HR = Admin 权限等价（仅差 superuser）
     ↓
无角色分配 API → 只能 Django Admin 操作 → 无审计追踪 → 合规违规
     ↓
CategoryListView IsAuthenticated 仅 → 任何员工可创建分类 → 数据完整性风险
     ↓
前端无 RoleGuard → 路由手动可达 → 组件渲染（信息泄露）
     ↓
AuditLog 缺系统操作 → 管理员行为不可追溯 → 合规风险（SOC 2 / ISO 27001）
```

### 1.3 现有代码脆弱性详情

#### F-01: IsHROrAdmin 权限类 — 二元闸门

**文件**: [backend/apps/core/permissions.py](backend/apps/core/permissions.py#L6-L12)

```python
class IsHROrAdmin(BasePermission):
    def has_permission(self, request, view):
        return request.user.is_authenticated and (
            getattr(request.user, "is_hr_admin", False) or request.user.is_superuser
        )
```

**脆弱性**:
- HR 和 Admin 仅有 `is_hr_admin` 一个布尔差异，`superuser` 是 Django 内置上帝模式
- 无法区分「HR 管理知识库」和「Admin 管理系统」的职责边界
- `getattr` 回退到 `False`，意味着属性不存在时静默拒绝而非报错

#### F-02: User 模型 — 无角色体系

**文件**: [backend/apps/users/models.py](backend/apps/users/models.py#L53)

```python
is_hr_admin = models.BooleanField(default=False)
```

**脆弱性**:
- `service_line` 和 `role_level` 是组织属性（部门/职级），不是功能角色
- 无 `Role` 模型、无 `Permission` 模型、无 `UserRole` 关联表
- `is_hr_admin` 是列级布尔，无法支持多角色或权限继承

#### F-03: CategoryListView — 严重权限缺口

**文件**: [backend/apps/knowledge/views.py](backend/apps/knowledge/views.py#L124)

```python
class CategoryListView(generics.ListCreateAPIView):
    serializer_class = DocumentCategorySerializer
    permission_classes = [permissions.IsAuthenticated]  # ← 任何登录用户
```

**脆弱性**:
- POST 方法无 `IsHROrAdmin` 闸门
- 任何 Employee 可创建文档分类，破坏知识库分类体系
- 无对象级权限，所有用户可浏览完整分类树

#### F-04: UserSerializer — 管理员名单泄露

**文件**: [backend/apps/users/serializers.py](backend/apps/users/serializers.py#L22)

```python
fields = ["id", "email", "username", ..., "is_hr_admin"]  # ← 公开暴露
```

**脆弱性**:
- `GET /api/v1/auth/me/` 返回当前用户的 `is_hr_admin` 状态
- 虽然仅返回自身状态，但此字段的存在表明系统以布尔标记管理员
- 攻击者可通过社工或观察 API 响应推断管理员身份

#### F-05: AuditLog — 系统操作黑洞

**文件**: [backend/apps/audit/models.py](backend/apps/audit/models.py#L12-L24)

```python
ACTION_CHOICES = [
    ("document_upload", "Document Upload"),
    ("document_delete", "Document Delete"),
    ...  # 仅 11 种内容操作
    # ← 缺: role_assign, role_revoke, user_create, config_change
]
```

**脆弱性**:
- 系统管理操作（角色分配、用户创建/停用、配置变更）无审计记录
- 管理员行为完全不可追溯
- SOC 2 Type II / ISO 27001 合规审计无法通过

### 2.2 新风险引入（需缓解措施）

| # | 新风险 | 严重度 | 缓解措施 | 责任 Phase |
|---|--------|--------|---------|-----------|
| 1 | **迁移期数据丢失** — `is_hr_admin=True` → UserRole 不完整 | 🔴 高 | Phase 2 双重授权窗口：`IsHROrAdmin` 同时检查新旧两套；数据迁移脚本验证 `UserRole.objects.filter(role__name='hr').count() == User.objects.filter(is_hr_admin=True).count()` | Phase 2 |
| 2 | **角色膨胀** — 模型支持任意角色，长期可能产生过多角色 | ⚠️ 中 | V4.0 仅 seed `hr` + `admin` 两个角色；角色创建仅 Admin 可操作（`user.assign_role` 权限）；建议设 `max_roles_per_user=3` 约束 | Phase 3 |
| 3 | **`is_superuser` 绕过** — Django superuser 默认绕过所有权限检查 | ⚠️ 中 | `HasPermission` 类显式处理 `is_superuser=True` 作为 override；文档化仅紧急访问使用；审计所有 superuser 登录；长期目标：限制 superuser 仅 Django Admin 紧急场景 | Phase 2 |
| 4 | **`has_permission()` 性能** — 每次请求查库，28+ 权限检查 | ⚠️ 低 | User 模型新增 `get_permissions()` 返回 set；在 request scope 内 memoize；或 JWT claims 缓存角色列表（但需处理角色变更后 token 过期问题） | Phase 3 |
| 5 | **JWT token 角色缓存不一致** — 角色变更后旧 token 仍有效 15 分钟 | ⚠️ 中 | 方案 B（推荐）：始终查库 + request-scoped 缓存，确保审计准确性。放弃 JWT claims 方案（A），因为无法保证变更后立即生效 | Phase 3 |
| 6 | **前端角色列表存储安全** — `roles: string[]` 在 localStorage | ⚠️ 低 | 前端仅用于路由守卫和 UI 渲染；所有 API 调用仍由后端 `HasPermission` 验证；前端 role 列表不可信，仅优化 UX | Phase 2 |

## 3. 架构方案：模型设计与权限执行

### 3.1 新建 `apps/rbac` — 4 个模型

> [来源: V3.8/项目功能模块汇总_V3.8.md §数据模型] — 当前仅有 User 模型，无角色/权限模型

| 模型 | 关键字段 | 说明 | DB 表名 |
|------|---------|------|---------|
| `Role` | `name` (hr/admin), `label`, `scope` (content/system), `is_active` | 功能角色定义，非组织 role_level | `rbac_role` |
| `Permission` | `codename` (document.upload), `resource`, `action`, `label` | 资源-操作级细粒度权限 | `rbac_permission` |
| `RolePermission` | `role` FK → Role, `permission` FK → Permission | 角色↔权限映射（多对多） | `rbac_rolepermission` |
| `UserRole` | `user` FK → User, `role` FK → Role, `assigned_by` FK → User, `is_active` | 用户↔角色分配，支持多角色，记录分配者 | `rbac_userrole` |

**模型设计细节**：

```python
# backend/apps/rbac/models.py

class Role(BaseModel):
    name = models.CharField(max_length=50, unique=True)      # 'hr', 'admin'
    label = models.CharField(max_length=100)                   # 'Knowledge Base Manager', 'System Super Admin'
    scope = models.CharField(max_length=20, choices=[
        ('content', 'Content'), ('system', 'System')
    ])
    is_active = models.BooleanField(default=True)

class Permission(BaseModel):
    codename = models.CharField(max_length=100, unique=True)   # 'document.create'
    resource = models.CharField(max_length=50)                  # 'document', 'category', 'user', 'audit', 'config'
    action = models.CharField(max_length=20)                    # 'create', 'read', 'update', 'delete', 'export'
    label = models.CharField(max_length=100)

    class Meta:
        unique_together = ['resource', 'action']

class RolePermission(BaseModel):
    role = models.ForeignKey(Role, on_delete=models.CASCADE, related_name='role_permissions')
    permission = models.ForeignKey(Permission, on_delete=models.CASCADE, related_name='granted_to_roles')

    class Meta:
        unique_together = ['role', 'permission']

class UserRole(BaseModel):
    user = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.CASCADE, related_name='user_roles')
    role = models.ForeignKey(Role, on_delete=models.PROTECT, related_name='assigned_users')
    assigned_by = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.SET_NULL, null=True, related_name='role_assignments')
    is_active = models.BooleanField(default=True)

    class Meta:
        unique_together = ['user', 'role']
```

### 3.2 User 模型变更

**新增方法**（在 [backend/apps/users/models.py](backend/apps/users/models.py) 中）：

```python
def has_role(self, role_name: str) -> bool:
    return UserRole.objects.filter(
        user=self, role__name=role_name, is_active=True
    ).exists()

def has_permission(self, codename: str) -> bool:
    return Permission.objects.filter(
        codename=codename,
        granted_to_roles__role__user_roles__user=self,
        granted_to_roles__role__user_roles__is_active=True,
    ).exists()

def get_permissions(self) -> set:
    """Return set of all active permission codenames for this user."""
    return set(
        Permission.objects.filter(
            granted_to_roles__role__user_roles__user=self,
            granted_to_roles__role__user_roles__is_active=True,
        ).values_list('codename', flat=True)
    )
```

**废弃 `is_hr_admin`**：Phase 2 保留列（双重授权窗口），Phase 4 移除列。

### 3.4 权限执行策略

**DRF Permission Classes**（主要执行层）：

| 新权限类 | 功能 | 替换目标 |
|----------|------|---------|
| `HasPermission` | 检查 `codename` 精确权限 | 替换所有 `IsHROrAdmin` [来源: V3.8/开发变更日志汇总_V3.8.md §permissions.py] |
| `HasRole` | 检查角色名 | 路由级粗粒度控制 |

```python
# backend/apps/core/permissions.py — 新增

class HasPermission(BasePermission):
    required_permission = None

    def has_permission(self, request, view):
        required = self.required_permission or getattr(view, 'required_permission', None)
        if not required:
            return True
        if not request.user.is_authenticated:
            return False
        if request.user.is_superuser:
            return True  # Django superuser bypass（仅紧急场景）
        return request.user.has_permission(required)

class HasRole(BasePermission):
    required_role = None

    def has_permission(self, request, view):
        required = self.required_role or getattr(view, 'required_role', None)
        if not required:
            return True
        if not request.user.is_authenticated:
            return False
        if request.user.is_superuser:
            return True
        return request.user.has_role(required)
```

**三层执行架构**：

1. **视图级** — `permission_classes = [IsAuthenticated, HasPermission]`，`required_permission = 'document.create'`
2. **方法级** — `get_permissions()` 方法分离 GET vs POST 权限
3. **Serializer 级** — `UserSerializer` 按请求者权限决定输出字段（`is_hr_admin` 移除，`roles` 仅管理员可见）

### 6.2 新增字段：`role_used`

```python
role_used = models.CharField(max_length=20, null=True, blank=True,
    help_text="Role used to authorize this action (for dual-role audit tracing)")
```

**用途**：当 Admin（同时拥有 hr + admin 双角色）执行内容操作时，记录使用的是 `hr` 还是 `admin` 角色，便于审计溯源。
