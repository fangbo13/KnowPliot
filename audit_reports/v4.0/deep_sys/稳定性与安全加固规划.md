## §1 XSS 攻击面分析与加固方案

### 1.1 ReactMarkdown href/src 未校验协议

**关联缺陷**: `[来源: V4.0/deep_sys_defect_list.md §DEFECT-002]`

**当前攻击面**:

| 白名单元素 | 风险属性 | 攻击向量 | 当前防护 | 缺失防护 |
|-----------|----------|----------|---------|---------|
| `a` | `href` | `javascript:alert(1)` | `target="_blank"` + `rel="noopener noreferrer"` | **href 协议校验** |
| `img` | `src` | `data:image/svg+xml,<svg onload=alert(1)>` | `loading="lazy"` | **src 协议校验** |
| 任何被禁元素 | children | `unwrapDisallowed={true}` 保留子文本 | 不执行脚本 | 子内容未消毒 |

**方案 A: 组件级 href/src 校验 (推荐 — 改动最小)**

```tsx
// MessageBubble.tsx — 自定义 a 组件增强协议校验
a: ({ href, children }) => {
  const safeProtocols = ['http://', 'https://', 'mailto:'];
  const isSafe = href && safeProtocols.some(p => href.toLowerCase().startsWith(p));
  if (!isSafe) {
    // 不安全 href → 仅渲染文本，不生成 <a> 标签
    return <span>{children}</span>;
  }
  return (
    <a href={href} target="_blank" rel="noopener noreferrer">{children}</a>
  );
},

img: ({ src, alt }) => {
  const safeProtocols = ['http://', 'https://'];
  const isSafe = src && safeProtocols.some(p => src.toLowerCase().startsWith(p));
  if (!isSafe) {
    // 不安全 src → 返回 alt 文本或空
    return alt ? <span>[{alt}]</span> : null;
  }
  return <img src={src} alt={alt || ''} loading="lazy" />;
},
```

**优点**: 改动仅限 `MessageBubble.tsx` 一个文件，不影响 ReactMarkdown 其他行为。
**缺点**: 不防护 Markdown AST 层面的其他攻击（如 `<details>` 内嵌恶意内容）。

**方案 B: DOMPurify 预处理 (更彻底 — 改动较大)**

```tsx
// 新增依赖: npm install dompurify @types/dompurify
import DOMPurify from 'dompurify';

// 在 ReactMarkdown 之前预处理
const sanitizedContent = DOMPurify.sanitize(message.content, {
  ALLOWED_TAGS: ALLOWED_ELEMENTS,
  ALLOWED_ATTR: ['href', 'src', 'alt', 'title', 'class'],
  FORBID_ATTR: ['onerror', 'onload', 'onclick'],
  FORCE_BODY: true,
});

<ReactMarkdown allowedElements={ALLOWED_ELEMENTS} unwrapDisallowed={true}>
  {sanitizedContent}
</ReactMarkdown>
```

**优点**: DOMPurify 是 OWASP 推荐的 HTML 消毒库，防护覆盖面最广。
**缺点**: 额外 npm 依赖 (~15KB gzipped)；DOMPurify 输出 HTML 字符串而非 Markdown AST，需确保 ReactMarkdown 正确处理消毒后的文本。

**推荐**: 先实施方案 A（紧急修复），再在后续版本实施方案 B（深度加固）。

### 1.2 localStorage JWT → httpOnly Cookie 迁移

**关联缺陷**: `[来源: V4.0/deep_sys_defect_list.md §DEFECT-003]`

**当前问题**:

```
当前流程:
  Login → POST /auth/login/ → {access, refresh} → 前端存 localStorage → XSS可窃取

目标流程:
  Login → POST /auth/login/ → 后端 Set-Cookie(httpOnly+Secure+SameSite=Strict)
  → 前端 localStorage 仅存 {user} (非敏感数据)
  → Axios/fetch 用 credentials:'include' 自动发送cookie
```

**后端改动**:
1. `users/views.py` LoginView: 登录成功后通过 `response.set_cookie()` 设置 httpOnly JWT cookies
2. `config/settings/base.py`: 添加 `JWT_COOKIE_SETTINGS` (httpOnly=True, Secure=True, SameSite='Strict')
3. DRF authentication: 添加 `JWTCookieAuthentication` 类从 cookie 读取 access token

**前端改动**:
1. `AuthProvider.tsx`: `login()` 不再存储 token 到 localStorage，仅存 `user` 对象
2. `client.ts`: Axios 拉取配置添加 `withCredentials: true`
3. `chatStore.ts`: `sendMessage` fetch 添加 `credentials: 'include'`
4. 401拦截器: 清除 cookie 调用 `POST /auth/logout/` (后端清除cookie)

**兼容性注意**: 迁移到 httpOnly cookie 后，DEFECT-011 (CSRF绕过) 变为真实威胁，必须同步添加 CSRF 保护（见§5.2）。

## §2 状态机竞态条件分析

### 2.1 StreamLifecycleManager TOCTOU — 实际安全但需文档化

**关联缺陷**: `[来源: V4.0/deep_sys_defect_list.md §DEFECT-006]`

**理论分析**:
```
createStreamAbortController(sessionId):
  Line 24: abortActiveStream()    ← 步骤A: 中断旧stream
  Line 26: controller = new AbortController()  ← 步骤B: 创建新controller
  Line 27: activeAbortController = controller   ← 步骤C: 赋值全局变量
  Line 28: activeStreamSessionId = sessionId    ← 步骤D: 赋值全局变量
```

**JS单线程安全性证明**: 步骤A→B→C→D在同一同步函数内执行。JS事件循环中，microtask/macrotask 不会在同步代码块中间插入。因此不存在真正的TOCTOU竞态。

**但存在架构耦合风险**: `streamPhase`（Zustand）和 `activeAbortController`（模块级变量）由两个不同系统管理。Zustand 的 `set()` 是同步的，但 Zustand 的 subscriber 通知是**异步的**（React batching）。这意味着：
- `lockSend()` → `set({isSendLocked: true})` → Zustand状态更新（同步）
- 但 React 组件看到 `isSendLocked=true` 是在下一次渲染（异步）

在 `lockSend()` 和 `createStreamAbortController()` 之间的**React batching窗口**内，UI 可能仍显示可发送状态。但 `sendMessage` 内部通过 `get().isSendLocked` 检查（读Zustand内部状态而非React渲染），所以实际防双击有效。

**建议**: 在 `StreamLifecycleManager.ts` 文件头部添加设计文档注释，明确声明：
```ts
/**
 * DESIGN NOTE: activeAbortController 和 activeStreamSessionId 是模块级单例变量，
 * 不存储在 Zustand 中（AbortController 不可序列化）。这意味着：
 * 1. 跨Tab不共享 — 需 BroadcastChannel 同步（见DEFECT-008）
 * 2. 与 streamPhase/isSendLocked 由不同机制管理 — 需手动确保一致性
 * 3. JS单线程保证 createStreamAbortController 内的abort→assign不会交错
 */
```

[来源: V3.4/bug_list.md §CRIT-001] + [来源: V3.5/reports/综合审计报告.md §streamPhase修复]

### 2.2 跨 Tab BroadcastChannel 同步方案

**关联缺陷**: `[来源: V4.0/deep_sys_defect_list.md §DEFECT-008]`

**设计**:

```ts
// 新文件: frontend/src/sync/crossTabSync.ts
const channel = new BroadcastChannel('ey-onboarding-sync');

// 发送端 — session切换/删除时广播
export function broadcastSessionSwitch(sessionId: string | null) {
  channel.postMessage({ type: 'session-switch', sessionId, timestamp: Date.now() });
}

export function broadcastSessionDelete(sessionId: string) {
  channel.postMessage({ type: 'session-delete', sessionId, timestamp: Date.now() });
}

// 接收端 — 其他Tab监听
channel.onmessage = (event) => {
  const { type, sessionId } = event.data;
  switch (type) {
    case 'session-switch':
      // 若当前Tab正在streaming旧session → abort
      if (getActiveStreamSessionId() !== sessionId) {
        abortActiveStream();
        resetTokenBatcher();
      }
      break;
    case 'session-delete':
      // 若当前Tab正在streaming被删除的session → abort + reset
      if (getActiveStreamSessionId() === sessionId) {
        abortActiveStream();
        resetTokenBatcher();
        useChatStore.getState().resetSession();
      }
      break;
  }
};
```

**集成点**:
- `chatStore.ts setActiveSession()` → 调用 `broadcastSessionSwitch(id)`
- `AppLayout.tsx handleDeleteSession()` → 调用 `broadcastSessionDelete(id)`

## §3 SSE 断连重连机制设计

**关联缺陷**: `[来源: V4.0/deep_sys_defect_list.md §DEFECT-007]`

### 3.1 方案对比

| 方案 | 优点 | 缺点 | 实现难度 | 推荐 |
|------|------|------|----------|------|
| A: 后端增量保存 | 断连后可恢复 | 每10 token写DB，DB压力大 | 高 | ❌ |
| B: 前端 sessionStorage | 无后端改动 | 仅保存前端可见内容，不保DB | 低 | ❌ |
| C: 混合方案 | citations事件含message_id，可查询 | 需新增API端点 | 中 | ✅ |

### 3.2 推荐方案: 混合重连 (方案C)

**设计原理**: 
1. 后端在 SSE `citations` 事件中发送 `message_id`（前端已有 `data.message_id` 在 `done` 事件中，但需提前到 `citations` 事件）
2. 前端在收到 `citations` 事件后，存储 `pendingMessageId`
3. 网络断开时，前端先调用 `GET /sessions/{id}/messages/` 查询是否有已完成的消息
4. 若找到 → 显示完整消息（从DB恢复）
5. 若未找到 → 显示断连提示 + "重试" 按钮

**后端改动** (views.py event_stream):
```python
# 在 citations 事件中预创建 assistant_message (status='streaming')
assistant_message = Message.objects.create(
    session=session, role="assistant", content="",
    status="streaming"  # ← 新字段
)
yield "event: citations\n"
yield f"data: {json.dumps({'message_id': str(assistant_message.id), **citations_data})}\n\n"

# 在 done 事件中更新 content + status='completed'
assistant_message.content = assistant_content
assistant_message.status = "completed"
assistant_message.save(update_fields=["content", "status", "token_count", ...])
```

**前端改动** (chatStore.ts sendMessage):
```ts
// 新增状态: pendingMessageId
case 'citations':
  data.message_id && set({ pendingMessageId: data.message_id });
  get().setStreamCitations(data);
  break;

// 断连恢复逻辑 (catch块):
catch (error) {
  if (networkError) {
    const pendingId = get().pendingMessageId;
    if (pendingId) {
      // 查询后端是否已保存完整消息
      try {
        const msgs = await chatApi.getMessages(sessionId);
        const completedMsg = msgs.find(m => m.id === pendingId);
        if (completedMsg) {
          // 恢复完整消息
          get().addMessage(completedMsg);
          get().finishStreamingMessage(completedMsg.id, sessionId);
          return true;  // 成功恢复
        }
      } catch {}
    }
    // 无法恢复 → 显示断连提示
    set({ streamPhase: 'error', sendError: 'error_network' });
  }
}
```

**Message 模型改动**: 需添加 `status` 字段 (`streaming/completed/failed`)。

## §4 JWT 刷新机制设计

**关联缺陷**: `[来源: V4.0/deep_sys_defect_list.md §DEFECT-009]`

### 4.1 静默刷新架构

```
当前流程 (断裂):
  Login → access_token(15min) → 使用 → 过期 → 401 → 强制登出

目标流程 (连续):
  Login → access_token(15min) + refresh_token(7day) → 使用
  → access接近过期(剩<30s) → 静默调用 POST /auth/token/refresh/
  → 获取新access+新refresh → 更新localStorage/cookie
  → 原请求继续（不中断）
```

### 4.2 前端实现

```ts
// client.ts — 添加 JWT 刷新拦截器
apiClient.interceptors.request.use(async (config) => {
  const { token, refreshToken } = getAuthTokens();
  if (!token) return config;

  // 检查 access token 是否即将过期
  const payload = decodeJWTPayload(token);
  const expiresAt = payload.exp * 1000;  // ms
  const now = Date.now();
  const bufferMs = 30000;  // 30秒缓冲

  if (expiresAt - now < bufferMs && refreshToken) {
    // 静默刷新
    try {
      const { data } = await axios.post('/api/v1/auth/token/refresh/', {
        refresh: refreshToken,
      });
      const newAuth = { token: data.access, refreshToken: data.refresh, user: getUser() };
      localStorage.setItem('ey-auth', JSON.stringify(newAuth));  // 或 httpOnly cookie
      config.headers.Authorization = `Bearer ${data.access}`;
    } catch {
      // refresh也过期 → 真正登出
      localStorage.removeItem('ey-auth');
      window.location.href = '/login';
    }
  } else {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});

// JWT payload解码 (不验证签名 — 仅读exp)
function decodeJWTPayload(token: string): { exp: number } {
  const base64 = token.split('.')[1];
  return JSON.parse(atob(base64));
}
```

### 4.3 SSE 流的 JWT 保护

**问题**: SSE 流建立后无法中途刷新 token（fetch 已发出，无法修改 headers）。

**方案**: 在 `sendMessage` 开始前检查 token 有效期：
```ts
// chatStore.ts sendMessage — SSE前置JWT检查
const token = getAuthToken();
const payload = decodeJWTPayload(token);
if (payload.exp * 1000 - Date.now() < 30000) {
  // Token即将过期 → 先刷新
  await refreshAuthToken();
  token = getAuthToken();  // 获取新token
}
```

## §5 CSRF 与限流加固方案

### 5.1 SSE 端点限流 — 紧急修复

**关联缺陷**: `[来源: V4.0/deep_sys_defect_list.md §DEFECT-001]`

**方案**: 添加 `@throttle_classes` 装饰器

```python
# views.py — send_message 添加限流
from rest_framework.throttling import UserRateThrottle

class SendMessageRateThrottle(UserRateThrottle):
    rate = '10/minute'  # 每用户每分钟10次（比一般API更严格，因为每次调用触发RAG+LLM）

@api_view(["POST"])
@permission_classes([permissions.IsAuthenticated])
@throttle_classes([SendMessageRateThrottle])
def send_message(request, session_id):
    ...
```

**rate 选择依据**: 
- 正常用户: 5-10 msg/hr = ~0.17 msg/min
- 活跃用户: 1-2 msg/min
- 防滥用: 10/min 足够覆盖正常使用，同时限制恶意高频调用
- DashScope qwen-plus 单次调用 ¥0.004 → 10/min = ¥0.04/min = ¥2.4/hr (可控)

**备选方案**: 转换为 `APIView` 类（自动继承 DRF throttle），但需要重构 SSE streaming 逻辑，改动较大。

### 5.2 CSRF 保护 — 显式声明 + 未来迁移准备

**关联缺陷**: `[来源: V4.0/deep_sys_defect_list.md §DEFECT-011]`

**当前状态**: `@api_view` 绕过 CsrfViewMiddleware。JWT header-based auth 下 CSRF 不是威胁（浏览器不自动发送 Authorization header）。

**方案 A: 显式声明安全原因 (当前推荐)**

```python
from django.views.decorators.csrf import csrf_exempt

@api_view(["POST"])
@permission_classes([permissions.IsAuthenticated])
@throttle_classes([SendMessageRateThrottle])
@csrf_exempt  # ← 显式声明：安全因为使用 header-based JWT auth
def send_message(request, session_id):
    """Send message with SSE streaming.
    
    CSRF exempt: This endpoint uses JWT Bearer token authentication.
    The token is sent via Authorization header, which browsers do not
    automatically include in cross-origin requests. CSRF protection
    is therefore not required for header-based auth.
    
    IMPORTANT: If JWT authentication migrates to httpOnly cookies,
    this exemption MUST be removed and CSRF token handling MUST be added.
    """
    ...
```

**方案 B: JWT→cookie 迁移后 (§1.2 完成后)**

迁移后浏览器自动发送 cookie，CSRF 成为真实威胁。需：
1. 移除 `@csrf_exempt`
2. 前端从 cookie 读取 CSRF token (`csrftoken` cookie)
3. SSE fetch 请求添加 `X-CSRFToken` header
4. Axios 添加 CSRF token interceptor

```ts
// client.ts — JWT→cookie后添加CSRF
apiClient.interceptors.request.use((config) => {
  const csrfToken = getCookie('csrftoken');
  if (csrfToken) {
    config.headers['X-CSRFToken'] = csrfToken;
  }
  return config;
});
```
