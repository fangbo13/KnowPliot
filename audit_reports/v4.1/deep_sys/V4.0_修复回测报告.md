## 2. 逐项详细结论

### 2.1 XSS 防护 (DEFECT-002) — 5 Payload 全部 PASS

**测试方法**: 通过容器内 Python 向 SSE 端点发送 5 种 XSS Payload，验证 DB 存储（原始文本）和前端渲染逻辑。

| Payload | DB 存储内容 | 前端处理逻辑 | 结果 |
|---|---|---|---|
| `<img src=x onerror=alert(1)>` | 原文存储 `'<img src=x onerror=alert(1)>'` | `img` 不在 ALLOWED_ELEMENTS → ReactMarkdown `unwrapDisallowed=true` 剳除 → 不渲染 | ✅ 纯文本，无执行 |
| `<script>alert(1)</script>` | 原文存储 | `script` 不在 ALLOWED_ELEMENTS → 完全剥离 | ✅ 纯文本，无脚本执行 |
| `<svg onload=alert(1)>` | 原文存储 | `svg` 不在 ALLOWED_ELEMENTS → 完全剥离 | ✅ 纯文本，无 onload |
| `[click](javascript:alert(1))` | 原文存储 | `a` 在白名单，但 `href` 协议校验：`SAFE_HREF_PROTOCOLS = ['http://', 'https://', 'mailto:']` → `javascript:` 不匹配 → 渲染为 `<span>` 纯文本 | ✅ 链接不可点击 |
| `[img](data:image/svg+xml,<svg onload=alert(1)>)` | 原文存储 | `img` src 校验：`SAFE_SRC_PROTOCOLS = ['http://', 'https://']` → `data:` 不匹配 → 渲染为 `[img]` 纯文本 | ✅ SVG 不加载 |

**结论**: V4.0 XSS 防护机制（ALLOWED_ELEMENTS 白名单 + 协议校验 + unwrapDisallowed）在 V4.1 中完全有效。后端存储原始文本（不做 HTML 转义），前端 ReactMarkdown 白名单机制确保所有危险标签和协议均被拦截。这是三层纵深防御的正确设计。[来源: V4.0/deep_sys_defect_list.md §DEFECT-002]

### 2.2 SSE 限流 (DEFECT-001) — PASS

**测试方法**: 容器内 Python 连续发送 5 次 POST `/api/v1/chat/sessions/{sid}/send/`，第 4 次触发 429。

**结果**:
- Request 1-3: HTTP 200（正常）
- Request 4: HTTP 429，响应体 `{"error":"Request was throttled. Expected available in 47 seconds.","detail":...}`
- `SendMessageRateThrottle` rate = `10/minute`，`DEFAULT_THROTTLE_RATES = {'user': '30/minute'}`

**说明**: 第 4 次即触发 429 是因为之前的测试请求仍在限流窗口内（60s 滑动窗口）。限流机制本身是正确的。

**结论**: V4.0 SSE 端点限流修复有效。`@throttle_classes([SendMessageRateThrottle])` 确保了 @api_view 端点不会绕过 DRF 默认限流。[来源: V4.0/deep_sys_defect_list.md §DEFECT-001]

### 2.3 并发状态修复 (DEFECT-006) — PASS (代码审计)

**代码审查结论**:

1. `StreamLifecycleManager.ts`: `createStreamAbortController(sessionId)` 调用流程：
   - `abortActiveStream()` → `new AbortController()` → `activeAbortController = controller` → `activeStreamSessionId = sessionId`
   - 全部同步操作，JS 单线程保证无 TOCTOU 竞态

2. `chatStore.ts`: `sendMessage()` 通过 `get()` 读取 `isSendLocked`（Zustand 内部状态同步读取），保证发送锁在 React batching 前生效

3. `crossTabSync.ts`: BroadcastChannel 实现 `session-switch` 和 `session-delete` 事件广播，其他 Tab 的活跃流会被正确 abort

**结论**: V4.0 并发状态修复（AbortController 生命周期 + isSendLocked + BroadcastChannel）在 V4.1 代码中完全保留且正确。[来源: V4.0/deep_sys_defect_list.md §DEFECT-006/008]

### 2.5 CSRF 防护 — PASS

**测试结果**: 无 JWT Token 的 POST 请求 → HTTP 401 `Authentication credentials were not provided`

**分析**: 本项目使用 JWT Token 认证（非 Django Session），DRF 的 JWT 认证绕过了 CsrfViewMiddleware 的 CSRF 检查。这是正确的设计——JWT 本身就是 CSRF 防护手段（Token 不存储在 Cookie 中，无法被跨站请求自动携带）。

**但存在风险**: `CORS_ALLOW_ALL_ORIGINS = True` 使得任何域名都可以发送携带 JWT 的跨域请求。如果 JWT 被 XSS 或存储泄露窃取，攻击者可以从任意域名发起 API 请求。（详见 SYS-V4.1-001）

### 2.6 输入清洗 — PASS

**SQL 注入 Payload**: `'; DROP TABLE users; --`
- DB 存储: 原文（`"'; DROP TABLE users; --"`）
- Django ORM 使用参数化查询，SQL 注入不可能通过 ORM 执行
- ChatMessageRequestSerializer 的 `content` 字段为 CharField(max_length=4000)，不做 HTML 转义（聊天场景的正确设计）

**路径穿越 Payload**: `../../../etc/passwd`
- DB 存储: 原文
- 后端无基于聊天内容的文件操作
- 无路径穿越风险

**结论**: 输入清洗策略正确——后端存储原文，前端渲染时转义/剥离危险内容。Django ORM 参数化查询天然防 SQL 注入。

### 2.7 500 响应泄露 (DEFECT-012) — PARTIAL PASS

**V4.0 修复**: `custom_exception_handler()` 对未处理异常返回 `{"error": "Internal server error"}`，不泄露 `str(exc)`。

**V4.1 验证**: DRF 视图（所有 API 端点）经过 `custom_exception_handler` 处理，确实不泄露异常详情。

**但**: `docker.py` 中 `DEBUG = True` 导致 Django 的默认 500 处理器返回完整 HTML 页面，包含：
- 完整 Python 堆栈跟踪
- 本地变量值
- SQL 查询
- Django settings 配置

这意味着：**非 DRF 视图的 500 错误（如 Django admin、静态文件错误）会泄露完整堆栈**。这是生产环境的严重安全缺陷。

**结论**: DEFECT-012 修复在 DRF 视图范围内有效，但 `DEBUG=True` 在更广泛范围内否定了这个修复的效果。

### 2.8 SSE 错误事件 (DEFECT-013) — PASS

**代码验证**: `chat/views.py` L218-222:
```python
except Exception as e:
    logger.error("Stream error for session %s: %s", session_id, e, exc_info=True)
    yield "event: error\n"
    yield f"data: {json.dumps({'error': 'stream_error'}, ensure_ascii=False)}\n\n"
```

SSE error 事件仅返回 `{"error": "stream_error"}`，完整异常信息仅记录在服务端日志。前端无法获取任何异常详情。

**结论**: V4.0 DEFECT-013 修复有效，SSE 错误事件不泄露异常信息。
