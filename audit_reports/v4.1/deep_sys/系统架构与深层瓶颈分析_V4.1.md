## 2. 竞态分析

### 2.1 前端并发竞态

**StreamLifecycleManager.ts 模块级单例**:

- `activeAbortController` 和 `activeStreamSessionId` 是模块级变量，非 Zustand 状态
- `createStreamAbortController(sessionId)` 的执行流程：`abortActiveStream()` → `new AbortController()` → 赋值
- **JS 单线程保证**: 这三步在同一个函数调用栈中，不会被其他事件交错。无 TOCTOU 竞态。[来源: V4.0/deep_sys_defect_list.md §DEFECT-006]

**跨 Tab 竞态**:

- BroadcastChannel 机制：Tab B 切换 session → 发送 `session-switch` → Tab A 收到 → abortActiveStream
- Tab B 删除 session → 发送 `session-delete` → Tab A 收到 → abortActiveStream + resetSession
- **竞态窗口**: BroadcastChannel 事件是异步的（约 10-50ms 延迟）。在极端场景下，Tab A 可能在收到 abort 前完成了一个 token 的 SSE 写入。但 chatStore 的 `finishStreamingMessage` 检查 `getActiveStreamSessionId()` 验证 session ID 一致性，防止数据污染。[来源: V4.0/deep_sys_defect_list.md §DEFECT-008]

**前端状态双轨问题**:

- `streamPhase/isSendLocked`（Zustand）与 `activeAbortController`（模块级）由不同机制管理
- Zustand `set()` 同步更新内部状态，但 React re-render 是异步的
- `sendMessage()` 通过 `get()` 读取 Zustand 内部状态（同步），因此 isSendLocked 在 React batching 前生效
- **潜在问题**: 如果两个 `sendMessage()` 在同一帧（rAF）中被调用，第一个的 `lockSend()` 状态可能还没反映到第二个的 `get()` 读取。但 JS 事件循环保证同一帧内的 Zustand `set()` 在 `get()` 之前完成。

### 2.2 后端并发竞态

**Django runserver 是单线程**: 每个请求由单一 WSGI 进程顺序处理。理论上不存在请求间竞态。

**但 SSE StreamingHttpResponse 是长连接**: 在 `event_stream()` generator 执行期间，Django runserver 可能无法处理新请求（取决于 server 实现）。

**实际风险**: 在 `runserver` 下，一个 SSE 流式响应可能阻塞后续请求直到完成。前端 AbortController 的 abort 信号会触发 `GeneratorExit`，释放 worker，但如果多个 SSE 请求并发到达，可能排队等待。

**Celery Worker 竞态**: `-c 4` 允许 4 个并发任务，但 `ingest_document` 任务对同一 Document 的 `status` 字段更新无原子保护：
```python
doc.status = "processing"
doc.save(update_fields=["status"])
# 如果同时触发两次 reindex，两个 worker 会同时将 status 设为 "processing"
# 然后 pipeline.ingest(doc) 会并行执行，可能导致重复 embedding
```
**无分布式锁**: 两次 `ingest_document.delay()` 对同一 document_id 会并行执行，无 Redis 锁保护。

## 3. 故障点标注

### 3.1 Redis 宕机

**影响范围**:

| 场景 | Celery 任务 | 前端 SSE | DRF 缓存 |
|---|---|---|---|
| Redis 宕机 | ❌ 任务无法投递/接收 | ⚠️ 不影响（SSE 不依赖 Redis） | ❌ throttle 缓存丢失 |
| Redis 恢复 | ✅ 自动重连 | ✅ 无影响 | ⚠️ throttle 窗口重置 |

**Celery 配置**:
- `CELERY_BROKER_URL = redis://redis:6379/0`
- Redis 无密码（`redis:7-alpine` 默认配置）
- Redis 无 `maxmemory` 配置（可能无限增长导致 OOM）
- `CELERY_RESULT_BACKEND = django-db`（任务结果存储在 PostgreSQL，不依赖 Redis 持久化）
- **关键**: `CELERY_ACCEPT_CONTENT = ["json"]` — 安全（防止 pickle 反序列化攻击）

**任务丢失风险**: Redis 无持久化配置（`save` 默认被 Alpine 镜像注释），宕机后未消费的任务会丢失。但由于 `CELERY_RESULT_BACKEND = django-db`，已完成任务结果不会丢失。

### 3.2 DB 连接池耗尽

**当前配置**:
- `CONN_MAX_AGE = 0` — **每个请求创建新连接，请求结束后关闭**
- `CONN_HEALTH_CHECKS = False` — 不检查连接健康状态
- 无连接池大小限制

**在 100 QPS 下**:
- 每秒 100 次连接创建 + 100 次连接关闭 = 200 次 TCP 操作/秒
- PostgreSQL 默认 `max_connections = 100`，但 Docker 容器配置未知
- 连接风暴风险：大量并发请求瞬间耗尽 PG 的 `max_connections`

**降级策略**: 无。`CONN_MAX_AGE = 0` 没有任何降级或连接复用机制。

**修复建议**: `CONN_MAX_AGE = 60`（复用连接 60s），`CONN_HEALTH_CHECKS = True`。

### 3.3 Worker 单点

**当前配置**:
- 1 个 Celery Worker 容器，`-c 4`（4 个并发 worker 进程）
- `CELERY_TASK_TIME_LIMIT = None` — 无超时限制
- `CELERY_TASK_SOFT_TIME_LIMIT = None` — 无软超时

**风险**:
- 如果一个 `ingest_document` 任务处理大文件（50MB PDF），可能运行数小时
- 4 个并发 slot 中 1 个被长时间占用，可用 slot 降至 3
- 如果 4 个都被大文件占用，后续任务排队无限等待
- 无 task routing：所有任务共享 4 slot，无优先级区分

**修复建议**: 添加 `CELERY_TASK_TIME_LIMIT = 300`（5 分钟硬超时），`CELERY_TASK_SOFT_TIME_LIMIT = 240`（4 分钟软超时）。

### 3.4 外部 API 依赖

**DashScope API**:
- `DASHSCOPE_API_KEY` 存储在 `.env` 文件中
- `SSL_VERIFY = False`（docker.py L27）— **禁用了 SSL 验证**，允许中间人攻击截获 API Key
- 无 fallback：DashScope 不可用时，RAG pipeline 完全失败
- 无 circuit breaker：连续失败会阻塞所有 chat 请求

## 4. 资源水位预估（100 QPS）

| 资源 | 当前配置 | 100 QPS 预估 | 风险 |
|---|---|---|---|
| CPU (Backend) | 单进程 runserver | ~80% (Django 无并发) | 🔴 阻断 |
| 内存 (Backend) | 无限制 | ~200MB + per-request overhead | 🟡 一般 |
| DB 连接 | CONN_MAX_AGE=0 | 100 连接/秒创建关闭 | 🔴 阻断 |
| Redis 连接 | 无密码，无 maxmemory | ~50 连接 | 🟡 一般 |
| Celery Worker | 1 worker, -c 4 | 4 concurrent max | 🔴 阻断（大文件占满） |
| DashScope API | 单 Key，无 rate limit | 100 call/min × ¥0.004 = ¥0.4/min | 🟡 一般 |
| 前端 JS Heap | MAX_ALL_MESSAGES=100 | ~2MB (100 messages) | 🟢 正常 |

**结论**: 系统在 100 QPS 下至少有 3 个阻断级瓶颈（CPU/DB连接/Worker），0 QPS → 100 QPS 的扩展需要：
1. 替换 runserver 为 gunicorn（多 worker）
2. 设置 CONN_MAX_AGE > 0（连接复用）
3. 增加 Celery worker 数量 + 设置 task timeout

## 5. 前端架构瓶颈

### 5.1 Zustand 状态管理

**内存管理**: `MAX_ALL_MESSAGES = 100`（V3.7 P1.3）限制了前端 JS Heap 增长。100 条消息 ≈ 50 轮对话，足够日常使用。

**StreamPhase 状态机**: `idle → connecting → searching → streaming → completing → error`。正确的状态机设计，每次转换都有明确的触发和清理逻辑。

### 5.2 React 渲染性能

**React.memo 优化**: `MemoizedMessageBubble` 使用自定义 comparator：
- Streaming 消息：每次都 re-render（content 变化）
- 非 Streaming 消息：仅在 id/content/isStreaming 变化时 re-render
- 消除了 O(n²) Markdown AST 解析

**TokenBatchRenderer**: rAF-based token batching 避免每 token 一次 setState。

### 5.3 前端安全架构

**XSS 三层纵深防御**:
1. `ALLOWED_ELEMENTS` 白名单（ReactMarkdown `allowedElements`）
2. `unwrapDisallowed = true`（不允许的元素被剥离而非保留）
3. 协议校验（`a` → SAFE_HREF_PROTOCOLS, `img` → SAFE_SRC_PROTOCOLS）

**JWT Token 存储**: `localStorage.setItem('ey-auth', JSON.stringify(newState))`
- **风险**: localStorage 中的 JWT token 可被 XSS 攻击窃取
- **缓解**: 当前 XSS 防护有效（三层纵深），但一旦 XSS 绕过，token 就会泄露
- **最佳实践**: 应将 token 存储在 httpOnly cookie 中（但会引入 CSRF 风险）
