## 三、前端渲染架构分析摘要

### 数据流向图

```
SSE tokens → TokenBatchRenderer (rAF合批) → chatStore (Zustand)
→ ChatPage (多selector订阅) → VirtualizedMessageList (react-virtuoso)
→ MemoizedMessageBubble → ReactMarkdown (非流式) / <span> (流式)
→ DOM
```

### StreamPhase 状态机

```
idle → connecting → searching → streaming → idle (完成)
                                              ↘ error (30s超时→100ms后回idle)
                                              ↘ abort (用户停止→截断保存→idle)
```

### 关键性能机制

| 机制 | 文件 | 效果 |
|------|------|------|
| rAF token batching | TokenBatchRenderer.ts | ~60次/秒 合批更新（vs 逐token ~数百次/秒） |
| 纯文本流式渲染 | MessageBubble.tsx:300-308 | 流式期间零Markdown AST开销 |
| react-virtuoso | VirtualizedMessageList.tsx | viewport+200px 外消息不渲染 |
| ref稳定性 | VirtualizedMessageList.tsx:56-59 | data引用流式期间稳定，仅活跃气泡重渲染 |
| IntersectionObserver创建控制 | ChatPage.tsx:132-148 | 避免流式期间60次/秒 IO重建 |

### 性能瓶颈标注

| 级别 | 瓶颈 | 文件 | 影响 |
|------|------|------|------|
| 🔴 HIGH | ReactMarkdown 初始解析阻塞 | MessageBubble.tsx:309-380 | 5000+字长回复主线程50-200ms |
| 🟡 MEDIUM | highlight.js 同步语法高亮 | MessageBubble.tsx:319 | 大代码块增加10-50ms |
| 🟡 MEDIUM | computeRounds O(n)扫描 | chatStore.ts | 100上限下可控 |
| 🟢 OPTIMIZED | rAF合批 | TokenBatchRenderer.ts | 渲染压力降低~80% |

[来源: V4.1/ui_ux/前端渲染与性能模块_V4.1.md]

## 四、环境限制说明

本次审计使用全新数据库（`pgdata_ui` volume），知识库为空（`Document.objects.count() = 0`），导致：
- AI 回复均为 RAG fallback 短文本，无法自然触发代码块/长文本场景
- 停止按钮瞬态无法通过截图捕获（回复<2s）
- 代码复制按钮无法通过真实代码块触发

后续审计建议导入知识库数据，实现完整场景覆盖。
