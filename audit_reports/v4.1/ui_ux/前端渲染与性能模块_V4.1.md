## 2. StreamPhase 状态机详细转换图

```
                    sendMessage() 调用
                         │
                         ▼
                   ┌──────────┐
                   │  idle    │ ←──── 初始状态 / 流式结束 / abort后
                   └─────┬────┘
                         │ set streamPhase='connecting'
                         │ lockSend()
                         ▼
                   ┌──────────┐
                   │connecting│ ←──── SSE fetch 开始
                   └─────┬────┘
                         │ HTTP响应头到达 或 3s定时器
                         │ set streamPhase='searching'
                         ▼
                   ┌──────────┐
                   │ searching│ ←──── 知识库检索阶段
                   └─────┬────┘
                         │ 首个token到达
                         │ set streamPhase='streaming'
                         ▼
                   ┌──────────┐  rAF合批更新streamContent
                   │ streaming│ ←──── AI回复流式输出
                   └─────┬────┘
                         │
          ┌──────────────┼──────────────┐
          │              │              │
          ▼              ▼              ▼
    ┌──────────┐   ┌──────────┐   流式完成
    │  error   │   │  abort   │   │
    │(超时/失败)│   │(用户停止)│   │
    └───┬──────┘   └───┬──────┘   ▼
        │              │      finishStreamingMessage()
        │ 100ms        │          → addMessage(完整内容+citations)
        │ 自动恢复      │          → streamContent=''
        ▼              ▼          → streamPhase='idle'
    ┌──────────┐   ┌──────────┐   → unlockSend()
    │  idle    │   │  idle    │
    └──────────┘   └──────────┘

    abort路径: 保留截断的streamContent → addMessage(截断消息)
```

## 3. 性能瓶颈标注

### 🔴 BOTTLENECK-1: ReactMarkdown 初始解析阻塞主线程

**位置**: `MessageBubble.tsx` lines 309-380
**场景**: 流式结束后，完整累积的 Markdown 字本一次性通过 `remark-gfm` → `rehype-highlight` → React 组件树生成
**影响**: 5000+ 字符的长回复可能导致主线程阻塞 50-200ms，造成短暂 UI 卡顿
**已有缓解**: 
  - 流式期间纯文本渲染（V3.7优化）
  - `memo` comparator 防止重解析
  - `ErrorBoundary` 防崩溃
**建议**: 可考虑分段 Markdown 渲染（先渲染前 2000 字，其余延迟），但当前 V3.7 优化已将此场景大幅缓解

### 🟡 BOTTLENECK-2: highlight.js 同步语法高亮

**位置**: `MessageBubble.tsx` line 319 (`rehypePlugins=[rehypeHighlight]`)
**场景**: 消息中包含多个代码块时，所有代码块在同一渲染周期内同步高亮
**影响**: 大代码块（>100行）的高亮时间为 O(code_length × vocabulary_size)，可能增加 10-50ms
**已有缓解**: 流式期间不触发高亮
**建议**: 可考虑 `requestIdleCallback` 延迟高亮，但当前规模下影响可控

### 🟡 BOTTLENECK-3: computeRounds O(n) 线性扫描

**位置**: `chatStore.ts` — `addMessage()` 和 `finishStreamingMessage()` 内
**场景**: 每次添加消息后执行 `computeRounds()` (O(n)) 和 `extractVisibleMessages()`
**影响**: 100条消息上限下影响可控，但无内存限制优化
**已有缓解**: `MAX_ALL_MESSAGES = 100` 硬上限
**建议**: 可考虑增量计算而非全量扫描

### 🟢 OPTIMIZED: rAF Token Batching (V3.7)

**位置**: `TokenBatchRenderer.ts`
**效果**: SSE 逐token → rAF合批 → ~60次/秒 Zustand更新（而非逐token ~数百次/秒）
**收益**: 流式期间渲染压力降低 ~80%

### 🟢 OPTIMIZED: Virtuoso ref稳定性 (V3.7)

**位置**: `VirtualizedMessageList.tsx` lines 56-59
**效果**: `streamContent`/`citations` 用 ref 而非 state → data引用流式期间稳定 → 不触发全列表重渲染
**收益**: 仅重渲染活跃流式气泡，其余 ~50条消息完全不动

### 🟢 OPTIMIZED: IntersectionObserver 创建控制 (V3.7)

**位置**: `ChatPage.tsx` lines 132-148
**效果**: IO 仅在 `messages.length` 变化时创建，不在 `streamContent` 变化时重建
**收益**: 避免流式期间 ~60次/秒 IO 创建销毁
