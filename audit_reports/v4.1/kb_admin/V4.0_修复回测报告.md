## 三、逐项结论

### reg_v4_kb_sec_01：垂直越权拦截（8/8 PASS）

**测试 Payload**：

```
Employee Token → 8 个端点测试：
GET  /api/v1/rbac/roles/          → 403 ✓
GET  /api/v1/rbac/permissions/    → 403 ✓
GET  /api/v1/rbac/user-roles/     → 403 ✓
POST /api/v1/rbac/user-roles/     → 403 ✓ (尝试给自己分配 admin 角色)
GET  /api/v1/rbac/users/          → 403 ✓
POST /api/v1/documents/           → 403 ✓ (IsHROrAdmin 闸门)
POST /api/v1/documents/categories/ → 403 ✓ (POST 需 IsHROrAdmin)
GET  /api/v1/documents/categories/ → 200 ✓ (GET 仅需 IsAuthenticated)
```

**结论**：V4.0 的垂直越权拦截完全有效。Employee 无法通过任何 Admin/RBAC 端点获取未授权数据。无 200 泄露。Admin 用户可正常访问所有 RBAC 端点（GET → 200）。

### reg_v4_kb_sec_02：CategoryListView POST 权限闸门（PASS）

**V4.0 修复**：`knowledge/views.py:129-137` 分离 GET/POST 权限，POST 需要 `IsHROrAdmin()`。

**回测结果**：
- Employee POST `/api/v1/documents/categories/` → 403 ✓
- HR POST `/api/v1/documents/categories/` → 201 ✓ (创建 category ID: 5039ad83-ef1c-4d6c-bc6d-0397c9916f10)

**结论**：修复有效。分类创建已受权限闸门保护。

### reg_v4_kb_flow_01：HR 上传与引用（PASS）

**回测结果**：
- HR GET `/api/v1/documents/` → 200 ✓
- HR GET `/api/v1/documents/categories/` → 200 ✓
- HR POST `/api/v1/documents/categories/` → 201 ✓
- HR GET `/api/v1/documents/templates/` → 200 ✓

**结论**：HR 对知识库的完整访问权限已保留。上传→引用链路可用。

### reg_v4_kb_jwt_01：JWT 过期与刷新（⚠ 部分通过，2 项新发现）

**回测结果**：
- Token refresh `/api/v1/auth/token/refresh/` → 200 ✓（可获取新 access token）
- JWT claims **包含 `is_hr_admin`** 但**不包含 `roles[]` 和 `permissions[]`** → 🔴 **新发现 KB-V4.1-009**
- Response body **包含 `is_hr_admin: True/False`** → 🔴 **P0-2 残留**
- Token blacklist **不生效**：logout 后同一 token 仍可访问 `/api/v1/auth/me/` → 200 → 🔴 **新发现 KB-V4.1-010**

**Admin JWT claims 解码**：
```json
{
  "is_hr_admin": true,
  "user_id": "aa0daa8f-d3e4-4a53-931a-8baa6b2b0efe"
  // roles: NOT_FOUND, permissions: NOT_FOUND
}
```

**HR JWT claims 解码**：
```json
{
  "is_hr_admin": true,
  "user_id": "..."
  // roles: NOT_FOUND, permissions: NOT_FOUND
}
```

**Employee JWT claims 解码**：
```json
{
  "is_hr_admin": false,
  "user_id": "..."
  // roles: NOT_FOUND, permissions: NOT_FOUND
}
```

**结论**：JWT refresh 机制正常，但 JWT payload 信息泄露（is_hr_admin）和 blacklist 失效是新发现的严重安全问题。

### reg_v4_kb_upload_01：文件类型校验（⚠ 部分通过，1 项新发现）

**回测结果**：
- `file_type=exe` → 被拒绝（`"exe" is not a valid choice`） ✓
- `file_type` 仅允许 `pdf, docx, html, txt` ✓
- **无 magic number 校验**：serializer 源码不含 `magic`, `content_type`, `validate` 关键词 → 🔴 **新发现 KB-V4.1-006**
- **无文件内容/哈希校验**：`file_hash` 字段存在但无自动填充逻辑

**Document FILE_TYPE_CHOICES**：
```
pdf: PDF
docx: Word Document
html: HTML
txt: Plain Text
```

**结论**：扩展名级别校验有效，但文件内容级别校验（magic number、MIME type）完全缺失。伪装扩展名的恶意文件可被接受。

## 四、总体结论

| 维度 | V4.0 修复是否有效 | 评分 |
|------|---|---|
| 垂直越权拦截 | ✓ 完全有效 | 10/10 |
| CategoryListView POST 闸门 | ✓ 完全有效 | 10/10 |
| HR 知识库访问 | ✓ 完全有效 | 10/10 |
| JWT 过期与刷新 | ⚠ 功能正常但安全缺陷 | 5/10 |
| 文件类型校验 | ⚠ 扩展名校验有效但无内容校验 | 5/10 |
| **综合** | **5/6 项有效，1 项部分有效** | **7.5/10** |

**回归结论**：V4.0 的核心权限修复（越权拦截、分类闸门、HR 访问）全部有效。但回测过程中发现 **3 项新安全缺陷**：

1. **JWT claims 泄露 `is_hr_admin`**（P0-2 残留，KB-V4.1-009）
2. **Token blacklist 失效**（KB-V4.1-010，logout 后 token 仍可用）
3. **无文件内容/magic number 校验**（KB-V4.1-006，伪装文件可入库）
