## 三、逐项结论

### SYS-V4.1-001: CORS 白名单
**结论**: ✅ CORS 白名单收紧有效。`http://evil.com` 被拒绝（无 Access-Control-Allow-Origin 头返回），`http://localhost:3030` 被接受（返回 `access-control-allow-origin: http://localhost:3030`）。CORS_ALLOW_ALL_ORIGINS 已彻底删除，不存在回归风险。

### SYS-V4.1-002: SafeErrorResponseMiddleware + ALLOWED_HOSTS
**结论**: ✅ ALLOWED_HOSTS 从 `["*"]` 收紧为 `["localhost","127.0.0.1","backend","0.0.0.0"]`，Host Header 注入风险消除。SafeErrorResponseMiddleware 类正确加载在 MIDDLEWARE 中。但需注意：docker.py 仍保留 `DEBUG=True`（SYS-V4.2-010 新缺陷），意味着非 DRF 中间件层的异常可能仍泄露堆栈信息，尽管 SafeErrorResponseMiddleware 覆盖了视图层异常。

### SYS-V4.1-003: SECRET_KEY 加固
**结论**: ✅ SECRET_KEY 67 字符（≥32 字节要求），三层逻辑不变。Auto-gen fallback 和 RuntimeWarning 机制完整。

### SYS-V4.1-004/005: Rate Limiting
**结论**: ✅ 双维度限流有效。AnonRateThrottle 100/min per IP + UserRateThrottle 30/min per user + LoginRateThrottle 5/min per IP。实测验证：第 6 次连续登录请求返回 429 Too Many Requests。

### SYS-V4.1-006: 并发竞态
**结论**: ✅ select_for_update() + transaction.atomic() + 409 Conflict 逻辑完整，防止并发 reindex 竞态。

### SYS-V4.1-009/010/011: Celery/Redis/SSL
**结论**: ✅ Celery 超时配置（300s hard/240s soft）不变；Redis requirepass + maxmemory 256mb 不变（无密码 PING 返回 NOAUTH）；SSL_VERIFY=True 不变（docker.py 无 override）。

### SYS-V4.1-007/008/012/013: 文档性/低风险修复
**结论**: ✅ SSE TODO 注释、Admin XSS escape、文件大小校验、JWT localStorage 风险备注均保持不变。

## 四、总体结论

**V4.1 全部 13 项修复在 V4.2 环境下回归通过，零回归项。** V4.0 全部 7 项遗留修复同样通过。系统安全基线维持在 V4.1 修复后的水平（预估 ~65/100）。

**需关注的新风险**：
- `docker.py` 中 `DEBUG=True` 虽然在 V4.1 时被有意保留（"dev priority"），但作为生产 Docker compose 的 settings 模块，仍存在信息泄露风险（SYS-V4.2-010 新缺陷）。
- IPv6 连接问题：测试发现 `localhost` 解析为 IPv6 `::1` 时，Django runserver 响应超时，必须使用 `127.0.0.1`（IPv4）才能正常连接。这暴露了 runserver 单线程阻塞的架构问题。
