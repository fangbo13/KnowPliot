## 二、P0 — 必须立即修复（阻断上线）

### FIX-001: JWT黑名单未阻断access token (SYS-V4.2-020)

**问题**: 调用 `/api/v1/auth/logout/` 后，被黑名单的access token仍可访问所有受保护API端点（返回200+用户数据），应返回401。

**根因**: `simplejwt` 的默认 `JWTAuthentication.authenticate()` 仅验证签名和过期时间，完全忽略 `blacklisted_tokens` 表。`BlacklistCheckingTokenRefreshSerializer` 仅在refresh流程中检查黑名单。

**修复方案**:

1. 新建文件 `backend/apps/users/authentication.py`:

```python
from rest_framework_simplejwt.authentication import JWTAuthentication
from rest_framework_simplejwt.token_blacklist.models import BlacklistedToken, OutstandingToken
from rest_framework.exceptions import AuthenticationFailed

class BlacklistCheckingJWTAuthentication(JWTAuthentication):
    """SYS-V4.2-020: Check blacklist table during access token validation."""
    def authenticate(self, request):
        result = super().authenticate(request)
        if result is None:
            return None
        user, validated_token = result
        jti = validated_token.get("jti")
        if jti:
            outstanding = OutstandingToken.objects.filter(jti=jti).first()
            if outstanding and BlacklistedToken.objects.filter(token=outstanding).exists():
                raise AuthenticationFailed("Token has been blacklisted.")
        return result
```

2. 修改 `backend/config/settings/base.py`:

```python
REST_FRAMEWORK = {
    "DEFAULT_AUTHENTICATION_CLASSES": [
        "apps.users.authentication.BlacklistCheckingJWTAuthentication",
    ],
}
```

3. 修改 `backend/apps/core/middleware.py` 中的 `AuthenticatedMediaMiddleware`，将 `JWTAuthentication()` 替换为 `BlacklistCheckingJWTAuthentication()`。

**验证**:
```bash
# 获取token
TOKEN=$(curl -s http://127.0.0.1:8030/api/v1/auth/token/ -X POST -H "Content-Type: application/json" -d '{"email":"admin@ey.com","password":"admin123"}' | python -c "import sys,json;print(json.load(sys.stdin)['access'])")
# logout
curl -s http://127.0.0.1:8030/api/v1/auth/logout/ -X POST -H "Authorization: Bearer $TOKEN"
# 期望: 401 Token has been blacklisted.
curl -s http://127.0.0.1:8030/api/v1/auth/me/ -H "Authorization: Bearer $TOKEN"
```

**文件**: `backend/apps/users/authentication.py` (新建), `backend/config/settings/base.py`, `backend/apps/core/middleware.py`
**工时**: ~2h

## 三、P1 — 严重缺陷（上线前必须修复）

### FIX-002: handleRetry绕过sendLock守卫 (UI-V4.2-002)

**问题**: `ChatPage.tsx` 中 `handleRetry` 直接调用 `sendMessage(lastUserMsg.content)`，绕过了 `handleSend` 的完整守卫链：`isStreaming || isSendLocked || isSendingRef.current || !navigator.onLine`。可能导致并发请求或离线无效请求。

**修复方案**:

修改 `frontend/src/pages/ChatPage.tsx`，将 `handleRetry` 重构为复用 `handleSend` 的守卫逻辑：

```typescript
// 将发送逻辑提取为带守卫的函数
const sendWithGuard = useCallback(async (content: string) => {
  if (isStreaming || isSendLocked || isSendingRef.current || !navigator.onLine) return;
  // ... 原 sendMessage 逻辑
}, [isStreaming, isSendLocked]);

// handleSend 和 handleRetry 都调用 sendWithGuard
const handleRetry = () => {
  if (lastUserMsg?.content) sendWithGuard(lastUserMsg.content);
};
```

**文件**: `frontend/src/pages/ChatPage.tsx`
**工时**: ~0.5天

### FIX-003: CrawlerAdminPage useEffect风暴 (UI-V4.2-001)

**问题**: `CrawlerAdminPage.tsx` 的 `useEffect(() => { fetchDocs(); }, [fetchDocs])` 依赖数组包含 `fetchDocs`（一个 `useCallback` 函数）。当 `fetchDocs` 的依赖 `t`（i18n）因语言切换/HMR重新创建时，整个effect重新触发，导致全量API请求风暴。

**修复方案**:

```typescript
// 改为空依赖数组 + polling管理
useEffect(() => {
  fetchDocs();
  const interval = setInterval(fetchDocs, 30000); // 30s polling
  return () => clearInterval(interval);
}, []); // 空依赖数组
```

**文件**: `frontend/src/pages/admin/CrawlerAdminPage.tsx`
**工时**: ~0.5天

### FIX-004: Admin Dashboard健康面板硬编码 (UI-V4.2-011)

**问题**: `AdminDashboardPage.tsx` 的 `loadSystemStatus` 调用 `/audit/` API 但完全不使用响应数据。所有状态字段硬编码为 `'running'`/`'connected'`，`total_documents: 0`。管理员看到虚假数据。

**修复方案**:

```typescript
const loadSystemStatus = async () => {
  try {
    const auditRes = await auditAPI.getStatus();
    // 使用真实API数据填充
    setSystemStatus({
      backend: auditRes.data?.backend_status || 'unknown',
      db: auditRes.data?.db_status || 'unknown',
      celery: auditRes.data?.celery_status || 'unknown',
      totalDocuments: auditRes.data?.total_documents || 0,
    });
  } catch {
    // 仅在API失败时显示unknown
    setSystemStatus({ backend: 'unknown', db: 'unknown', celery: 'unknown', totalDocuments: 0 });
  }
};
```

**文件**: `frontend/src/pages/admin/AdminDashboardPage.tsx`
**工时**: ~0.5天

### FIX-005: ErrorBoundary重试丢失全部SPA状态 (UI-V4.2-010)

**问题**: `ErrorBoundary.tsx` 的Retry按钮执行 `window.location.reload()`，杀死全部Zustand状态、SSE连接，重置SPA到初始加载状态。

**修复方案**:

```typescript
// 改为setState重挂载子树
handleRetry = () => {
  this.setState({ hasError: false, error: null });
};
```

**文件**: `frontend/src/components/ErrorBoundary.tsx`
**工时**: ~0.5天

### FIX-006: admin账号无法访问Admin/Knowledge页面

**问题**: UAT测试中 `admin@ey.com` 访问 `/admin/dashboard` 和 `/admin/knowledge` 被RoleGuard重定向到 `/chat`。

**修复方案**: 确认 `admin@ey.com` 的User模型 `role_level` 字段设置为 `'admin'`。如果是数据库问题，运行：
```bash
docker compose exec backend python manage.py shell -c "
from apps.users.models import User
u = User.objects.get(email='admin@ey.com')
print(f'role_level={u.role_level}')
u.role_level = 'admin'
u.save()
print('Updated to admin')
"
```

**文件**: 数据库数据（非代码）
**工时**: ~0.1天

### FIX-008: Sidebar hover --color-fill未定义 (UI-V4.2-007)

**问题**: `globals.css` 中 `.sidebar-session-item:hover` 使用 `background: var(--color-fill)` 但该变量未在 `:root` 或 `[data-theme="dark"]` 中定义。

**修复**: 在 `frontend/src/styles/globals.css` 的 `:root` 中添加：
```css
--color-fill: rgba(0, 82, 255, 0.04);
--color-fill-secondary: rgba(0, 82, 255, 0.08);
```
在 `[data-theme="dark"]` 中添加：
```css
--color-fill: rgba(77, 124, 255, 0.08);
--color-fill-secondary: rgba(77, 124, 255, 0.12);
```

**工时**: ~0.1天

### FIX-009: Google Fonts CDN Docker环境加载失败

**问题**: Docker容器内无法加载 `fonts.googleapis.com` 字体（net::ERR_NETWORK_ACCESS_DENIED）。

**修复**: 在 `frontend/Dockerfile` 或nginx配置中添加本地字体fallback，或在 `index.html` 中添加 `font-display: swap`。

**工时**: ~0.5天（可选）

### FIX-010: NetworkStatusBanner animation引用未定义keyframe (UI-V4.2-012)

**问题**: `animation: 'slideDown 0.3s ease-out'` 引用 `@keyframes slideDown` 但该keyframe未在globals.css中定义。

**修复**: 移除inline animation，依赖antd Alert内置动画；或在globals.css中定义 `@keyframes slideDown`。

**工时**: ~0.1天

### FIX-011: emoji非屏幕阅读器友好 (UI-V4.2-008)

**问题**: `MessageBubble.tsx` 中引用计数使用 `📎` emoji作为前缀，屏幕阅读器无法正确朗读。

**修复**: 替换为 `<PaperClipOutlined aria-label={t('sources')} />` antd icon。

**工时**: ~0.1天

### FIX-012: 字数计数器无ARIA live region (UI-V4.2-009)

**问题**: `{inputValue.length}/4000` 字数计数器无 `aria-live` 或 `role="status"`。

**修复**:
```html
<div role="status" aria-live="polite" aria-label={`Character count: ${inputValue.length} of 4000`}>
```

**工时**: ~0.1天
