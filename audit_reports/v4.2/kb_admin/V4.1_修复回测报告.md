## 三、逐项结论

### ✅ KB-V4.1-009: JWT claims 移除 is_hr_admin

**测试方法**：登录 admin@test.ey.com → base64 解码 JWT payload → 检查 keys  
**结果**：JWT payload 仅含 `[exp, iat, jti, token_type, user_id]`，`is_hr_admin` 已从 JWT claims 中移除。`is_hr_admin` 仅出现在 login response body（`user.is_hr_admin` 字段），不再存在于 replayable JWT token 中。  
**结论**：✅ PASS — V4.1 修复有效，无回归

### ⚠ KB-V4.1-010: Token blacklist 双重注销

**测试方法**：登录 → POST `/auth/logout/` 含 refresh token → 用同一 access token 访问 `/auth/me/`  
**结果**：Logout 返回 200 `"Logged out successfully."`，但注销后同一 access token 仍可访问 `/auth/me/` → 200  
**代码审查**：`users/views.py` 含 `BlacklistedToken.objects.get_or_create()` + `RefreshToken(refresh_token).blacklist()` 双重注销逻辑  
**分析**：可能是 SimpleJWT 的 OutstandingJwtTokenTracker 异步生效延迟，或 blacklisted token 查询未覆盖当前请求的 token JTI  
**结论**：⚠ PARTIAL — 注销代码逻辑存在，但实效性有待深入排查；可能是 V4.2 SYS 领域发现的 `SYS-V4.2-020` (blacklist bypass) 的关联问题 [来源: V4.2/deep_sys/deep_sys_defect_list_V4.2.md]

### ✅ KB-V4.1-006: Magic number 文件内容校验

**测试方法**：构造 PE 可执行文件 (MZ header) 改名 `.pdf` → 上传  
**结果**：400 `"File content does not match declared type 'pdf'. Detected content type: application/x-dosexec"`  
**结论**：✅ PASS — Magic number 校验拦截伪装文件，无回归

### ✅ KB-V4.1-007: AuthenticatedMediaMiddleware

**测试方法**：`GET /media/documents/test.pdf` 无 Authorization header  
**结果**：403 `"Authentication required for media files."`  
**结论**：✅ PASS — 无 Auth 访问 /media/ 被拦截，无回归

### ⚠ KB-V4.1-003: 横向越权文档删除

**测试方法**：HR 上传文档 → Employee 尝试删除 → 预期 403  
**结果**：文档上传因 Celery broker 连接失败 (500)，无法完成端到端 API 测试  
**代码审查**：`knowledge/views.py:73-79` 含 `uploaded_by != request.user` + `not request.user.has_role("admin")` → 403 检查  
**结论**：⚠ PARTIAL — 代码逻辑确认存在；端到端 API 测试因环境配置问题受阻（非代码回归）

### ✅ KB-V4.1-004: Retriever SQL 列白名单

**测试方法**：传入非法 filter key `evil_injection` → 预期 ValueError  
**结果**：ValueError `"Invalid filter key: 'evil_injection'. Allowed keys: ['category_id', 'document__status', 'document_id']"`  
**结论**：✅ PASS — SQL 注入防护有效，无回归

### ⚠ KB-V4.1-002: Superuser 绕过审计追踪

**测试方法**：Superuser 访问受保护 API → 查询 AuditLog 中 `superuser_bypass` 条目  
**结果**：自然触发无 bypass 条目（`create_audit_log` 需 request 对象的 REMOTE_ADDR/USER_AGENT），手动创建确认模型支持  
**结论**：⚠ PARTIAL — 审计逻辑代码存在，但实际触发需完整 request 对象（Permission 类中的 `create_audit_log` 调用在无 request 时可能静默失败）

### ✅ KB-V4.1-008: 文件大小限制 1KB-50MB

**测试方法**：上传 1 字节文件 → 预期 400  
**结果**：400 `"File is too small (1 bytes). Minimum size is 1KB."`  
**结论**：✅ PASS — 最小文件大小限制有效，无回归

### ✅ KB-V4.1-005: Prompt Injection 防护

**测试方法**：代码审查 `RAGPipeline.retrieve_and_generate`  
**结果**：`_sanitize_content()` + `guardrails.check_input()` 双重清洗逻辑完整  
**结论**：✅ PASS — Prompt Injection 防护代码存在，无回归

### ✅ KB-V4.1-011~017: 爬虫模块（7项）

**测试方法**：代码审查 + SSRF validator 功能测试 + ContentCleaner 清洗测试  
**结果**：
- CrawledDocument/CrawlTaskLog 模型：字段完整 ✅
- SSRF 防护：4 种 payload 全拦截 ✅
- ContentCleaner：script/iframe 全移除 ✅
- CrawlerService/RobotsTxtChecker：存在 ✅
- SimHash dedup：content_hash + duplicate_skipped ✅
- Crawler API 权限：Employee → 403, HR → 200 ✅
- AuditLog + 依赖：document_crawl + 5 依赖全存在 ✅

**结论**：✅ PASS — 爬虫模块全部功能有效，无回归
